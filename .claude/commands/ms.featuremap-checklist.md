# /ms.featuremap-checklist - Independent PRD Checklist

Dispatch the **featuremap-checklist-author** subagent (`.claude/agents/`) to
read the source PRD set only and create an independent commitment checklist
for later PRD-to-Feature-Map verification.

This command is intentionally upstream of `/ms.pre-verify`. The author never
reads `docs/prd/feature-map.md` — an extraction anchored by the decomposition
it will later be compared against is worthless as a baseline. Cross-vendor
independence is preserved downstream: `/ms.pre-verify`'s Layer-2 station has
Codex and Antigravity read the PRDs directly, so the same-model authorship of
this baseline is a conscious, documented trade (2026-07-19 — this station was
previously a background Codex run; the isolated-subagent form is faster,
non-flaky, and keeps memory isolation, while vendor diversity moved to the
verification layer where it matters).

The baseline author also remains blind to Audit signals, audit-tier policy, and
tier receipts. Tier metadata must not contaminate PRD-only extraction.

## Usage

```bash
/ms.featuremap-checklist @docs/prd/PRD.md [@docs/prd/another.md]
```

Execution is a foreground subagent dispatch; in a conducted run
(`/ms.pre-specter`) it may run concurrently with `/ms.featuremap`, since both
read only the PRD set.

## Output

The author must write:

```text
docs/prd/featuremap-checklist.md
```

(Legacy path from the Codex era: `docs/prd/codex/checklist.md` — consumers and
`specter-gate.sh` read the new path first and fall back to the legacy path, so
established projects keep working without regeneration.)

## Execution Steps

### Step 0: Resolve Source PRDs

Use the PRD paths supplied in `$ARGUMENTS`.

If no PRD paths are supplied, read `docs/prd/feature-map.md` only to discover
the source PRD path list, not to inspect Feature content or ownership. If that
is not possible, stop and ask the user for the PRD paths.

The **author** must not read (blindness contract — enforced by the agent
definition and restated in the dispatch prompt):

- `docs/prd/feature-map.md` Feature sections or PRD Commitment Index
- `docs/prd/feature-map.checklist.md`
- `docs/prd/checklists/*`
- `specs/*`, `plan.md`, `tasks.md`

**Session read policy**: per AGENTS.md §2 — reuse files already read this session; a fresh `Read` immediately before `Edit`/`Write` is still required.

### Step 1: Dispatch The Isolated Author

Dispatch the **featuremap-checklist-author** subagent with **file paths only**
(AGENTS §2 dispatch discipline — never paste PRD content):

```text
You are the featuremap-checklist-author. Extract the independent PRD-only
commitment checklist.

Source PRDs (read each IN FULL; read nothing else):
<insert PRD path list>

Output file: docs/prd/featuremap-checklist.md

Extract every durable PRD commitment: functional requirements, user journeys,
acceptance criteria, constraints, non-functional requirements, integration
promises, data/storage/migration/privacy/logging/audit/security promises,
explicit exclusions or deferred work, and behavior-constraining product
principles. Assign stable IDs C001, C002, ... For each commitment include the
PRD reference, commitment type, concise summary, and how it must appear in a
Feature Map.

Rules: do not invent Features or decide ownership; do not merge separate
commitments; do not drop small, operational, cross-cutting, or negative
requirements; record ambiguities as their own flagged rows; ENGLISH only.

Use exactly this structure:

# PRD-to-FeatureMap Checklist

**Mode**: prd-only
**Generated By**: featuremap-checklist-author (isolated subagent)
**Source PRDs**:
- <path>

## Commitment Inventory

| ID | PRD Ref | Type | Requirement / Commitment | Must Appear In Feature Map As |
| --- | --- | --- | --- | --- |

## Acceptance Criteria Inventory

| ID | PRD Ref | Acceptance Criterion | Expected Feature Map Handling |
| --- | --- | --- | --- |

## Cross-Cutting Concerns

| ID | PRD Ref | Concern | Expected Ownership Rule |
| --- | --- | --- | --- |

## Data / Migration / Integration Promises

| ID | PRD Ref | Promise | Expected Feature Map Handling |
| --- | --- | --- | --- |

## Explicit Exclusions / Deferred Work

| ID | PRD Ref | Exclusion Or Deferred Work | Expected Feature Map Handling |
| --- | --- | --- | --- |

## Ambiguities

| ID | PRD Ref | Ambiguity | Why It Matters For Feature Mapping |
| --- | --- | --- | --- |

## Slicing Risks

| Risk | Why It Matters | What To Check In Feature Map |
| --- | --- | --- |

## Verification Checklist

- [ ] Every commitment above has exactly one owning Feature.
- [ ] Every acceptance criterion is represented in a Feature done criterion or test expectation.
- [ ] Every explicit exclusion has a destination Feature.
- [ ] Every cross-cutting concern has one real owner, with earlier stubs only if needed.
- [ ] Every data, migration, and integration promise is assigned to a Feature.
- [ ] No user-facing behavior is diluted into vague implementation wording.
```

### Step 2: Deterministic Artifact Check (host)

After the subagent returns, verify the artifact mechanically: the file exists,
is non-empty, and contains `**Mode**: prd-only` plus at least one `C###` row.
If the check fails, re-dispatch the author once (fresh); if it still fails,
stop and report — never hand-write the baseline yourself, and never let
`/ms.pre-verify` run against a missing baseline.

### Step 2.5: Run-State Ledger (bookkeeping, not a gate)

Append one line to `.specify/specter-run.jsonl` (create it if needed;
append-only, never rewritten — a missing/corrupt ledger never blocks this
command), with `verdict` set to `PASS` when Step 2's artifact check passed and
`FAIL` otherwise. (Legacy note: the Codex-era background version recorded
`PENDING`; conductors still accept a legacy `PENDING` entry with an existing
artifact as satisfied when resuming old ledgers.)

```bash
mkdir -p .specify
printf '{"ts":"%s","cycle":"pre","feature":null,"step":"featuremap-checklist","verdict":"%s","artifacts":["docs/prd/featuremap-checklist.md"]}\n' \
  "$(date -u +%Y-%m-%dT%H:%M:%SZ)" "<PASS|FAIL>" >> .specify/specter-run.jsonl
```

### Step 3: Report

Display in Korean:

```text
독립 PRD 체크리스트 작성을 완료했습니다. (featuremap-checklist-author 서브에이전트)

📄 산출물: docs/prd/featuremap-checklist.md (<N> commitments)
🎯 다음 단계: /ms.pre-verify
```
