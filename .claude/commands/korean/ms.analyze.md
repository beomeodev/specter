# /ms.analyze - 문서 일관성 + TRUST 유효성 검사

작업이 사양 요구 사항과 일치하는지 확인한 다음 점진적인 3단계 TRUST 품질 검사를 수행합니다.

## 개요

이 명령은 **2단계 유효성 검사 프로세스**를 수행합니다.

**1단계: 문서 일관성** (`/speckit.analyze`를 통해)

- tasks.md가 모든 spec.md 요구 사항을 충실히 구현하는지 확인합니다.
- 누락된 요구 사항, 불일치, 분리된 작업을 확인합니다.
- **불일치가 발견되면 차단**

**2단계: TRUST 3단계 유효성 검사** (My-Spec 추가)

- **레벨 1**: 구조 확인 - 치명적인 경우 차단
- **레벨 2**: 품질 확인 - 치명적인 경우 차단
- **레벨 3**: 심층 분석 - 경고만

**실행 모드**:

- **엄격 모드** (기본값): 치명적인 위반은 즉시 차단
- **보고서 모드** (`--report`): 모든 레벨을 실행하고 모든 문제를 한 번에 보고

## 실행 단계

### 0단계: 프로젝트 컨텍스트 로드

**프로젝트 문서 자동 로드**:
- `.specify/memory/constitution.md` (헌법 - 필수)
- `AGENTS.md` (AI 지침, 코딩 표준 - 있는 경우)
- `specs/[spec-id]/spec.md` (기능 사양 - 필수)
- `specs/[spec-id]/tasks.md` (작업 목록 - 필수)

**헌법, spec.md 또는 tasks.md가 없는 경우**:
- 오류 표시: "필수 파일이 없습니다. 먼저 `/ms.init`, `/ms.specify` 및 `/ms.tasks`를 실행하십시오."
- 종료

**분석 참조**:
- 헌법 섹션 II (단순성 우선 - 파일 크기 제한: ≤500 SLOC)
- 헌법 섹션 V (TRUST 원칙 - 테스트 커버리지 ≥85%, 보안, 가독성)
- 헌법 섹션 IX (프로젝트별 제약 조건 - **있는 경우**, `/ms.constitution`에 의해 추가됨)
- AGENTS.md (코딩 표준, 복잡성 제한 ≤10 - 있는 경우)

**이 문서들은 프로젝트별 규칙을 적용하기 위해 TRUST 유효성 검사를 안내합니다.**

### 1단계: 문서 일관성 확인

spec-tasks 일관성을 위해 `/speckit.analyze` 실행:

```
/speckit.analyze
```

이것은 다음을 확인합니다.

- 모든 spec.md 기능 요구 사항(FR)에 해당하는 작업이 있습니다.
- 분리된 작업 없음(사양 요구 사항이 없는 작업)
- 작업 설명이 사양 요구 사항을 정확하게 반영합니다.
- 구현 순서가 종속성을 따릅니다.

**불일치가 발견된 경우**:

```
❌ 문서 일관성 실패

Tasks.md가 spec.md 요구 사항을 완전히 구현하지 않습니다.

누락된 요구 사항:
- FR-3: 암호 재설정 기능
- FR-7: 속도 제한

분리된 작업:
- T023: Redis 캐싱 추가(해당 FR 없음)

- tasks.md를 업데이트하고 /ms.analyze를 다시 실행하십시오.
```

**종료**: 코드 1 (TRUST 유효성 검사를 진행할 수 없음)

**일관된 경우**: 성공을 표시하고 1.5단계로 진행

### 1.5단계: 적응형 TRUST 분석(정량적 결정)

**1단계: 프로젝트 크기 측정(필수)**

```bash
# 소스 코드 라인 수 계산(주석, 공백 제외)
TOTAL_SLOC=$(find src/ -name "*.ts" -o -name "*.py" -o -name "*.js" 2>/dev/null | xargs wc -l 2>/dev/null | tail -1 | awk '{print $1}')

# 소스 파일 수 계산
FILE_COUNT=$(find src/ -name "*.ts" -o -name "*.py" -o -name "*.js" 2>/dev/null | wc -l)

# --report 플래그 확인
REPORT_MODE=$(echo "$ARGUMENTS" | grep -c "\-\-report")
```

**2단계: 의사 결정 트리 적용**

우선순위 순서대로 실행(첫 번째 일치에서 중지):

```
┌─────────────────────────────────────────────────────────────┐
│ 의사 결정 트리(우선순위 순서)                               │
├─────────────────────────────────────────────────────────────┤
│ 1. IF REPORT_MODE = 1                                       │
│    → 복잡함(병렬로 3개 에이전트 모두)                      │
│                                                              │
│ 2. IF TOTAL_SLOC < 500 OR FILE_COUNT < 10                   │
│    → 간단함(순차적 TRUST, 현재 동작)                        │
│                                                              │
│ 3. IF TOTAL_SLOC < 2000 OR FILE_COUNT < 50                  │
│    → 보통(에이전트 2개)                                      │
│                                                              │
│ 4. IF TOTAL_SLOC ≥ 2000 OR FILE_COUNT ≥ 50                  │
│    → 복잡함(에이전트 3개)                                      │
│                                                              │
│ 5. FALLBACK (측정 불가)                                     │
│    → 간단함(안전한 기본값, 순차적)                          │
└─────────────────────────────────────────────────────────────┘
```

**3단계: 하위 에이전트 전략 실행**

위에서 결정된 복잡성에 따라:

**간단한 경우**:
  - 하위 에이전트 0개 → 순차적 TRUST 유효성 검사(현재 동작)
  - 2단계로 바로 진행

**보통인 경우**:
  - 병렬로 2개의 하위 에이전트 시작:
    1. **trust-validator 에이전트(레벨 2)**:
       ```
       "TRUST 레벨 1 및 레벨 2 확인 실행"
       ```

    2. **tag-auditor 에이전트**:
       ```
       "프로젝트 전체에서 TAG 무결성 확인"
       ```

**복잡한 경우**:
  - 병렬로 3개의 하위 에이전트 시작:
    1. **trust-validator 에이전트(레벨 1)**:
       ```
       "TRUST 레벨 1 구조 확인 실행"
       ```

    2. **trust-validator 에이전트(레벨 2)**:
       ```
       "TRUST 레벨 2 품질 확인 실행"
       ```

    3. **trust-validator 에이전트(레벨 3)**:
       ```
       "TRUST 레벨 3 심층 분석 실행"

       참고: 레벨 3에는 TAG 유효성 검사가 포함되지만 상세한 TAG 보고서를 위해 tag-auditor를 별도로 실행할 수 있습니다.
       ```

**중요**: 항상 병렬로 에이전트를 시작합니다(여러 작업 호출이 있는 단일 메시지).

**디버그 출력**(투명성을 위해):
```json
{
  "complexity_metrics": {
    "total_sloc": 1500,
    "file_count": 35,
    "report_mode": false
  },
  "decision": "MODERATE",
  "reason": "규칙 3: TOTAL_SLOC < 2000",
  "agents_spawned": 2
}
```

### 1.6단계: TRUST 결과 종합

**하위 에이전트가 시작된 경우**:
- 모든 에이전트의 위반 사항 병합
- 심각도순으로 정렬(치명적 → 높음 → 중간 → 낮음)
- 레벨당 위반 횟수 계산
- 구현이 차단되었는지 확인(치명적인 항목이 발견된 경우)
- 통합된 TRUST 보고서 생성

**그렇지 않은 경우**:
- 순차 실행 결과 사용

### 2단계: TRUST 유효성 검사(3단계)

**1단계(문서 일관성)를 통과한 경우에만 실행됩니다.**

**하위 에이전트가 시작된 경우(1.5단계)**:
- 1.6단계의 종합된 결과 사용
- 순차 실행 건너뛰기
- 통합 보고서 표시

**그렇지 않은 경우**(단순 경로 - 순차 실행):

#### 레벨 1: 구조 확인

**확인**:

-   ✅ `tests/` 디렉토리 존재(치명적)
-   ✅ `.gitignore`에 `.env` 포함(치명적)
-   ✅ 파일 ≤500 SLOC(중간 - 코드 파일만, 문서는 제외)

**실행**(엄격 모드):

```
⚙️ 레벨 1: 구조 확인

  tests/ 디렉토리 확인 중... ✅
  .gitignore 확인 중... ✅
  파일 크기 확인 중... ⚙️ (경과 시간: 2초)
```

-   **치명적인 위반이 발견된 경우**: 즉시 종료하고 위반 사항을 표시하고 수정 명령을 제공합니다.
-   **통과한 경우**: 레벨 2로 진행
-   **시간 초과 없음** - 완료될 때까지 실행

#### 레벨 2: 품질 확인

**프로젝트 유형 감지**:

```bash
detect_project_type() {
  [ -f "package.json" ] && echo "typescript" && return
  [ -f "pyproject.toml" ] || [ -f "requirements.txt" ] && echo "python" && return
  [ -f "go.mod" ] && echo "go" && return
  [ -f "Cargo.toml" ] && echo "rust" && return
  echo "unknown"
}
```

**확인**:

-   ✅ 테스트가 성공적으로 실행됨(치명적)
-   ✅ 린팅이 경고 없이 통과(치명적)
-   ✅ 유형 검사가 통과(치명적)

**실행**(엄격 모드):

```
⚙️ 레벨 2: 품질 확인

  테스트 실행 중... (경과 시간: 15초)
  ✅ 테스트 통과(47초)

  린터 실행 중... (경과 시간: 2초)
  ✅ 린팅 통과

  유형 검사기 실행 중... (경과 시간: 8초)
  ✅ 유형 검사 통과
```

-   **레벨 1을 통과한 경우에만 실행**
-   **치명적인 위반이 발견된 경우**: 종료하고 위반 사항을 표시하고 수정 명령을 제공합니다.
-   **통과한 경우**: 레벨 3으로 진행
-   **시간 초과 없음** - 진행률 표시와 함께 완료될 때까지 실행

**프로젝트 유형 감지 실패**:

```
⚠️ 중간: 프로젝트 유형을 감지할 수 없습니다.

언어별 구성이 없습니다.
- package.json (TypeScript/JavaScript)
- pyproject.toml 또는 requirements.txt (Python)
- go.mod (Go)
- Cargo.toml (Rust)

레벨 2 품질 확인(테스트, 린팅, 유형 검사)을 건너뜁니다.
레벨 3(TAG 무결성)으로 진행합니다.
```

#### 레벨 3: 심층 분석

**확인**:

-   ✅ 커버리지 ≥85%(높음 - 경고만)
-   ✅ 함수당 복잡성 ≤10(높음 - 경고만)
-   ✅ 보안 스캔 통과(중간 - 권고)
-   ✅ TAG 무결성(분리/중복에 대해 치명적, 끊어진 체인에 대해 높음/중간)

**TAG 무결성**:

```bash
validate_tag_integrity() {
  # 모든 @SPEC, @TEST, @CODE 태그 찾기
  rg '@(SPEC|TEST|CODE):[A-Z0-9]+-[0-9]{3}' -oN src tests | sort -u > .specify/tags.txt

  # 분리된 TAG 확인(해당 파일 없음)
  # 중복 TAG ID 확인
  # 끊어진 SPEC→TEST→CODE 체인 확인
}
```

**실행**(엄격 모드):

```
⚙️ 레벨 3: 심층 분석

  TAG 스캔 중... (경과 시간: 5초)
  ✅ TAG 스캔 완료(12초)

  커버리지 계산 중... (경과 시간: 20초)
  ⚠️ 커버리지: 78% (목표: 85%)
```

-   **레벨 2를 통과한 경우에만 실행**
-   **높음/중간/낮음 위반**: 경고를 표시하고 계속합니다.
-   **시간 초과 없음** - 진행률 표시와 함께 완료될 때까지 실행

## TRUST 원칙

상세한 TRUST 원칙은 [헌법 섹션 V](../../.specify/memory/constitution.md)를 참조하십시오.

**유효성 검사 범위:**
- **T** - 테스트 우선: 커버리지 ≥85%, 테스트 성공적으로 실행
- **R** - 가독성: 파일 ≤500 SLOC, 함수 ≤100 LOC, 복잡성 ≤10
- **U** - 통일성: 엄격한 유형 지정
- **S** - 보안: 입력 유효성 검사, `.gitignore`에 `.env` 포함
- **T** - 추적 가능: 100% TAG 무결성

## 점진적 실행 규칙

```
레벨 1 (구조)
  ↓
  치명적인 경우 → 수정 지침과 함께 종료
  ↓
레벨 2 (품질)
  ↓
  치명적인 경우 → 수정 지침과 함께 종료
  ↓
레벨 3 (심층)
  ↓
  높음/중간/낮음 → 경고 표시, 계속
```

## 고급 기능

### 보고서 모드

조기 종료 없이 모든 레벨을 실행한 다음 포괄적인 보고서를 표시합니다.

```bash
/ms.analyze --report
```

**동작**:

-   치명적인 위반이 발견되더라도 모든 레벨을 계속 진행합니다.
-   레벨 1, 2, 3의 모든 문제를 수집합니다.
-   마지막에 포괄적인 보고서를 표시합니다.

**예시**:

```
⚙️ 2단계: TRUST 유효성 검사(보고서 모드)

  레벨 1: 구조 확인...
    ❌ 치명적: tests/ 디렉토리를 찾을 수 없습니다.
    ❌ 치명적: .gitignore에 .env가 없습니다.

  레벨 2: 품질 확인... (레벨 1 실패에도 불구하고 계속)
```

## 오류 처리

-   **LEVEL1_CRITICAL**: 구조 위반 발견

    -   예: "tests/ 디렉토리 누락"
    -   수정: `mkdir tests`

-   **LEVEL2_CRITICAL**: 품질 위반 발견

    -   예: "5개의 경고와 함께 린팅 실패"
    -   수정: `npm run lint --fix`

-   **LEVEL3_WARNINGS**: 심층 분석 문제

    -   예: "커버리지 78% < 85%"
    -   제안: "커버되지 않은 코드에 대한 테스트 추가"

-   **PROJECT_TYPE_NOT_DETECTED**: 언어를 확인할 수 없음
    -   심각도: 중간
    -   조치: 레벨 2 건너뛰고 레벨 3으로 계속

**2단계(TRUST 유효성 검사)에 치명적인 위반이 있는 경우**:

3. 치명적인 문제가 해결될 때까지 `/ms.implement`로 진행할 수 없음

## 참고

-   **2단계 프로세스**: 문서 일관성 우선, 그 다음 TRUST 유효성 검사
-   **1단계는 필수**: TRUST 유효성 검사가 실행되기 전에 통과해야 함
-   **점진적 실행**: TRUST 레벨은 이전 레벨이 통과하는 경우에만 실행됨(엄격 모드)
-   **시간 초과 없음**: 5초마다 진행률 표시와 함께 완료될 때까지 실행
-   **실행 모드**: 엄격(기본값, 조기 종료) 또는 보고서(모든 문제 보기)
-   **차단 동작**: 치명적 = 진행할 수 없음, 높음/중간 = 경고만
-   **수정 지침**: 각 위반에는 특정 수정 명령이 포함됨
-   **경고 추적**: 높은 경고는 `.specify/warnings.log`에 기록됨
-   **TAG 무결성**: 추적 가능성에 중요하며 TRUST 레벨 3에서 시행됨
-   **사용자 제어**: `--skip-tests`, `--skip-coverage`, `--fast` 옵션 사용 가능
