## 사용자 입력

```text
$ARGUMENTS
```

진행하기 전에 사용자 입력을 **반드시** 고려해야 합니다(비어 있지 않은 경우).

## 개요

`.specify/memory/constitution.md`에서 프로젝트 헌법을 업데이트하고 있습니다. 이 파일은 대괄호 안에 자리 표시자 토큰(예: `[PROJECT_NAME]`, `[PRINCIPLE_1_NAME]`)이 포함된 템플릿입니다. 귀하의 임무는 (a) 구체적인 값을 수집/파생하고, (b) 템플릿을 정확하게 채우고, (c) 종속 아티팩트 전체에 모든 수정 사항을 전파하는 것입니다.

다음 실행 흐름을 따르십시오.

1.  `.specify/memory/constitution.md`에서 기존 헌법 템플릿을 로드합니다.
    *   `[ALL_CAPS_IDENTIFIER]` 형식의 모든 자리 표시자 토큰을 식별합니다.
    **중요**: 사용자는 템플릿에 사용된 원칙보다 적거나 많은 원칙을 요구할 수 있습니다. 숫자가 지정된 경우 이를 존중하고 일반 템플릿을 따릅니다. 그에 따라 문서를 업데이트합니다.

2.  자리 표시자에 대한 값 수집/파생:
    *   사용자 입력(대화)이 값을 제공하는 경우 이를 사용합니다.
    *   그렇지 않으면 기존 리포지토리 컨텍스트(README, 문서, 포함된 경우 이전 헌법 버전)에서 유추합니다.
    *   거버넌스 날짜의 경우: `RATIFICATION_DATE`는 원래 채택 날짜입니다(알 수 없는 경우 묻거나 TODO로 표시). `LAST_AMENDED_DATE`는 변경 사항이 있는 경우 오늘 날짜이고, 그렇지 않으면 이전 날짜를 유지합니다.
    *   `CONSTITUTION_VERSION`은 시맨틱 버전 관리 규칙에 따라 증가해야 합니다.
        *   MAJOR: 이전 버전과 호환되지 않는 거버넌스/원칙 제거 또는 재정의.
        *   MINOR: 새로운 원칙/섹션 추가 또는 실질적으로 확장된 지침.
        *   PATCH: 명확화, 표현, 오타 수정, 비의미적 개선.
    *   버전 범프 유형이 모호한 경우 최종 결정 전에 근거를 제안합니다.

3.  업데이트된 헌법 내용 초안 작성:
    *   모든 자리 표시자를 구체적인 텍스트로 바꿉니다(프로젝트에서 아직 정의하지 않기로 선택한 의도적으로 유지된 템플릿 슬롯을 제외하고 대괄호로 묶인 토큰이 남지 않음 - 남은 항목은 명시적으로 정당화).
    *   제목 계층 구조를 유지하고 주석은 여전히 명확한 지침을 추가하지 않는 한 교체되면 제거할 수 있습니다.
    *   각 원칙 섹션이 간결한 이름 줄, 협상 불가능한 규칙을 포착하는 단락(또는 글머리 기호 목록), 명확하지 않은 경우 명시적인 근거를 갖도록 합니다.
    *   거버넌스 섹션에 수정 절차, 버전 관리 정책 및 규정 준수 검토 기대치가 나열되도록 합니다.

4.  일관성 전파 체크리스트(이전 체크리스트를 활성 유효성 검사로 변환):
    *   `.specify/templates/plan-template.md`를 읽고 "헌법 확인" 또는 규칙이 업데이트된 원칙과 일치하는지 확인합니다.
    *   범위/요구 사항 정렬을 위해 `.specify/templates/spec-template.md`를 읽습니다. 헌법이 필수 섹션 또는 제약 조건을 추가/제거하는 경우 업데이트합니다.
    *   `.specify/templates/tasks-template.md`를 읽고 작업 분류가 새롭거나 제거된 원칙 기반 작업 유형(예: 관찰 가능성, 버전 관리, 테스트 규율)을 반영하는지 확인합니다.
    *   `.specify/templates/commands/*.md`의 각 명령 파일을 읽고(이 파일 포함) 일반적인 지침이 필요할 때 오래된 참조(CLAUDE와 같은 에이전트별 이름만)가 남아 있지 않은지 확인합니다.
    *   런타임 지침 문서(예: `README.md`, `docs/quickstart.md` 또는 에이전트별 지침 파일(있는 경우))를 읽습니다. 변경된 원칙에 대한 참조를 업데이트합니다.

5.  동기화 영향 보고서 생성(업데이트 후 헌법 파일 상단에 HTML 주석으로 추가):
    *   버전 변경: 이전 → 신규
    *   수정된 원칙 목록(이름이 변경된 경우 이전 제목 → 새 제목)
    *   추가된 섹션
    *   제거된 섹션
    *   업데이트가 필요한 템플릿(✅ 업데이트됨 / ⚠ 보류 중) 파일 경로 포함
    *   의도적으로 연기된 자리 표시자가 있는 경우 후속 TODO

6.  최종 출력 전 유효성 검사:
    *   설명되지 않은 대괄호 토큰이 남아 있지 않음.
    *   버전 줄이 보고서와 일치함.
    *   날짜 ISO 형식 YYYY-MM-DD.
    *   원칙은 선언적이고 테스트 가능하며 모호한 언어가 없음("해야 한다" → 적절한 경우 MUST/SHOULD 근거로 대체).

7.  완성된 헌법을 `.specify/memory/constitution.md`에 다시 씁니다(덮어쓰기).

8.  사용자에게 다음을 포함하는 최종 요약 출력:
    *   새 버전 및 범프 근거.
    *   수동 후속 조치를 위해 플래그가 지정된 모든 파일.
    *   제안된 커밋 메시지(예: `docs: amend constitution to vX.Y.Z (principle additions + governance update)`).

서식 및 스타일 요구 사항:

-   템플릿과 정확히 동일한 마크다운 제목을 사용합니다(수준을 강등/승격하지 않음).
-   가독성을 유지하기 위해 긴 근거 줄을 줄 바꿈합니다(이상적으로 100자 미만). 그러나 어색한 줄 바꿈으로 강제로 적용하지는 않습니다.
-   섹션 사이에 한 줄의 공백을 유지합니다.
-   후행 공백을 피합니다.

사용자가 부분 업데이트(예: 단일 원칙 수정만)를 제공하는 경우에도 유효성 검사 및 버전 결정 단계를 수행합니다.

중요한 정보가 누락된 경우(예: 비준 날짜를 정말로 알 수 없는 경우) `TODO(<FIELD_NAME>): 설명`을 삽입하고 연기된 항목 아래의 동기화 영향 보고서에 포함합니다.

새 템플릿을 만들지 마십시오. 항상 기존 `.specify/memory/constitution.md` 파일에서 작업하십시오.
