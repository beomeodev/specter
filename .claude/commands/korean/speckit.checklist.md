## 체크리스트 목적: "영어를 위한 단위 테스트"

**핵심 개념**: 체크리스트는 **요구 사항 작성을 위한 단위 테스트**입니다. 즉, 특정 도메인의 요구 사항의 품질, 명확성 및 완전성을 검증합니다.

**검증/테스트용이 아님**:

- ❌ "버튼이 올바르게 클릭되는지 확인"
- ❌ "오류 처리가 작동하는지 테스트"
- ❌ "API가 200을 반환하는지 확인"
- ❌ 코드/구현이 사양과 일치하는지 확인

**요구 사항 품질 검증용**:

- ✅ "모든 카드 유형에 대해 시각적 계층 구조 요구 사항이 정의되어 있습니까?" (완전성)
- ✅ "'눈에 띄는 표시'가 특정 크기/위치로 정량화되어 있습니까?" (명확성)
- ✅ "모든 대화형 요소에서 호버 상태 요구 사항이 일관되게 정의되어 있습니까?" (일관성)
- ✅ "키보드 탐색에 대한 접근성 요구 사항이 정의되어 있습니까?" (적용 범위)
- ✅ "로고 이미지를 로드하지 못할 때 발생하는 상황을 사양에서 정의합니까?" (엣지 케이스)

**비유**: 사양이 영어로 작성된 코드라면 체크리스트는 단위 테스트 스위트입니다. 구현이 작동하는지 여부가 아니라 요구 사항이 잘 작성되고 완전하며 명확하고 구현 준비가 되었는지 테스트하는 것입니다.

## 사용자 입력

```text
$ARGUMENTS
```

진행하기 전에 사용자 입력을 **반드시** 고려해야 합니다(비어 있지 않은 경우).

## 실행 단계

1.  **설정**: 리포지토리 루트에서 `.specify/scripts/bash/check-prerequisites.sh --json`을 실행하고 FEATURE_DIR 및 AVAILABLE_DOCS 목록에 대한 JSON을 구문 분석합니다.
    *   모든 파일 경로는 절대 경로여야 합니다.
    *   "I'm Groot"와 같은 인수에 작은따옴표가 있는 경우 이스케이프 구문을 사용합니다(예: 'I'\''m Groot' 또는 가능한 경우 큰따옴표: "I'm Groot").

2.  **의도 명확화(동적)**: 최대 3개의 초기 컨텍스트 명확화 질문을 도출합니다(미리 준비된 카탈로그 없음). 다음을 **반드시** 수행해야 합니다.
    *   사용자의 표현 + 사양/계획/작업에서 추출한 신호에서 생성됩니다.
    *   체크리스트 내용을 실질적으로 변경하는 정보에 대해서만 질문합니다.
    *   `$ARGUMENTS`에서 이미 명확한 경우 개별적으로 건너뜁니다.
    *   너비보다 정밀도를 선호합니다.

    생성 알고리즘:
    1.  신호 추출: 기능 도메인 키워드(예: 인증, 지연 시간, UX, API), 위험 지표("중요", "필수", "규정 준수"), 이해 관계자 힌트("QA", "검토", "보안 팀") 및 명시적 결과물("a11y", "롤백", "계약").
    2.  신호를 관련성 순으로 순위가 매겨진 후보 초점 영역(최대 4개)으로 클러스터링합니다.
    3.  명시적이지 않은 경우 예상 대상 및 시기(작성자, 검토자, QA, 릴리스)를 식별합니다.
    4.  누락된 차원 감지: 범위 너비, 깊이/엄격함, 위험 강조, 제외 경계, 측정 가능한 수용 기준.
    5.  이러한 원형에서 선택한 질문을 공식화합니다.
        *   범위 구체화(예: "X 및 Y와의 통합 접점을 포함해야 합니까, 아니면 로컬 모듈 정확성으로 제한해야 합니까?")
        *   위험 우선순위 지정(예: "이러한 잠재적 위험 영역 중 어느 것이 필수 게이팅 검사를 받아야 합니까?")
        *   깊이 보정(예: "이것은 가벼운 사전 커밋 온전성 목록입니까, 아니면 공식 릴리스 게이트입니까?")
        *   대상 프레이밍(예: "작성자만 사용합니까, 아니면 PR 검토 중에 동료가 사용합니까?")
        *   경계 제외(예: "이번 라운드에서 성능 튜닝 항목을 명시적으로 제외해야 합니까?")
        *   시나리오 클래스 격차(예: "복구 흐름이 감지되지 않았습니다. 롤백/부분 실패 경로가 범위에 있습니까?")

    질문 서식 규칙:
    *   옵션을 제시하는 경우 옵션 | 후보 | 중요한 이유 열이 있는 간결한 테이블을 생성합니다.
    *   최대 A-E 옵션으로 제한하고 자유 형식 답변이 더 명확한 경우 테이블을 생략합니다.
    *   사용자가 이미 말한 내용을 다시 말하도록 요청하지 마십시오.
    *   투기적 범주를 피하십시오(환각 없음). 불확실한 경우 명시적으로 질문하십시오. "X가 범위에 속하는지 확인하십시오."

    상호 작용이 불가능할 때의 기본값:
    *   깊이: 표준
    *   대상: 코드 관련인 경우 검토자(PR), 그렇지 않으면 작성자
    *   초점: 상위 2개 관련성 클러스터

    질문(Q1/Q2/Q3 레이블)을 출력합니다. 답변 후: 2개 이상의 시나리오 클래스(대체/예외/복구/비기능 도메인)가 불분명한 경우 각각 한 줄의 정당화(예: "미해결 복구 경로 위험")와 함께 최대 2개의 더 구체적인 후속 질문(Q4/Q5)을 할 수 있습니다. 총 5개의 질문을 초과하지 마십시오. 사용자가 명시적으로 더 이상 거부하면 에스컬레이션을 건너뜁니다.

3.  **사용자 요청 이해**: `$ARGUMENTS` + 명확화 답변 결합:
    *   체크리스트 테마 도출(예: 보안, 검토, 배포, ux)
    *   사용자가 언급한 명시적인 필수 항목 통합
    *   초점 선택을 범주 스캐폴딩에 매핑
    *   사양/계획/작업에서 누락된 컨텍스트 유추(환각하지 않음)

4.  **기능 컨텍스트 로드**: FEATURE_DIR에서 읽기:
    *   spec.md: 기능 요구 사항 및 범위
    *   plan.md(있는 경우): 기술적 세부 정보, 종속성
    *   tasks.md(있는 경우): 구현 작업

    **컨텍스트 로드 전략**:
    *   활성 초점 영역과 관련된 필요한 부분만 로드(전체 파일 덤핑 방지)
    *   긴 섹션을 간결한 시나리오/요구 사항 글머리 기호로 요약하는 것을 선호
    *   점진적 공개 사용: 격차가 감지된 경우에만 후속 검색 추가
    *   소스 문서가 큰 경우 원시 텍스트를 포함하는 대신 중간 요약 항목 생성

5.  **체크리스트 생성** - "요구 사항에 대한 단위 테스트" 만들기:
    *   `FEATURE_DIR/checklists/` 디렉토리가 없으면 만듭니다.
    *   고유한 체크리스트 파일 이름 생성:
        *   도메인을 기반으로 짧고 설명적인 이름 사용(예: `ux.md`, `api.md`, `security.md`)
        *   형식: `[domain].md`
        *   파일이 있는 경우 기존 파일에 추가
    *   CHK001부터 순차적으로 항목 번호 매기기
    *   각 `/speckit.checklist` 실행은 새 파일을 만듭니다(기존 체크리스트를 덮어쓰지 않음).

    **핵심 원칙 - 구현이 아닌 요구 사항 테스트**:
    모든 체크리스트 항목은 다음을 위해 요구 사항 자체를 평가해야 합니다.
    *   **완전성**: 필요한 모든 요구 사항이 있습니까?
    *   **명확성**: 요구 사항이 명확하고 구체적입니까?
    *   **일관성**: 요구 사항이 서로 일치합니까?
    *   **측정 가능성**: 요구 사항을 객관적으로 확인할 수 있습니까?
    *   **적용 범위**: 모든 시나리오/엣지 케이스가 처리됩니까?

    **범주 구조** - 요구 사항 품질 차원별로 항목 그룹화:
    *   **요구 사항 완전성**(필요한 모든 요구 사항이 문서화되었습니까?)
    *   **요구 사항 명확성**(요구 사항이 구체적이고 명확합니까?)
    *   **요구 사항 일관성**(요구 사항이 충돌 없이 정렬됩니까?)
    *   **수용 기준 품질**(성공 기준이 측정 가능합니까?)
    *   **시나리오 적용 범위**(모든 흐름/사례가 처리됩니까?)
    *   **엣지 케이스 적용 범위**(경계 조건이 정의되어 있습니까?)
    *   **비기능적 요구 사항**(성능, 보안, 접근성 등 - 지정되어 있습니까?)
    *   **종속성 및 가정**(문서화되고 검증되었습니까?)
    *   **모호성 및 충돌**(무엇을 명확히 해야 합니까?)

    **체크리스트 항목 작성 방법 - "영어를 위한 단위 테스트"**:

    ❌ **잘못됨**(구현 테스트):
    *   "방문 페이지에 에피소드 카드 3개가 표시되는지 확인"
    *   "데스크톱에서 호버 상태가 올바르게 작동하는지 테스트"
    *   "로고 클릭이 홈으로 이동하는지 확인"

    ✅ **올바름**(요구 사항 품질 테스트):
    *   "추천 에피소드의 정확한 수와 레이아웃이 지정되어 있습니까?" [완전성]
    *   "'눈에 띄는 표시'가 특정 크기/위치로 정량화되어 있습니까?" [명확성]
    *   "모든 대화형 요소에서 호버 상태 요구 사항이 일관되게 정의되어 있습니까?" [일관성]
    *   "모든 대화형 UI에 대해 키보드 탐색 요구 사항이 정의되어 있습니까?" [적용 범위]
    *   "로고 이미지를 로드하지 못할 때 대체 동작이 지정되어 있습니까?" [엣지 케이스]
    *   "비동기 에피소드 데이터에 대한 로딩 상태가 정의되어 있습니까?" [완전성]
    *   "사양에서 경쟁 UI 요소에 대한 시각적 계층 구조를 정의합니까?" [명확성]

    **항목 구조**:
    각 항목은 이 패턴을 따라야 합니다.
    *   요구 사항 품질에 대해 묻는 질문 형식
    *   사양/계획에 작성된(또는 작성되지 않은) 내용에 집중
    *   대괄호 안에 품질 차원 포함 [완전성/명확성/일관성 등]
    *   기존 요구 사항을 확인할 때 사양 섹션 `[Spec §X.Y]` 참조
    *   누락된 요구 사항을 확인할 때 `[Gap]` 마커 사용

    **품질 차원별 예시**:

    완전성:
    *   "모든 API 실패 모드에 대해 오류 처리 요구 사항이 정의되어 있습니까? [Gap]"
    *   "모든 대화형 요소에 대해 접근성 요구 사항이 지정되어 있습니까? [완전성]"
    *   "반응형 레이아웃에 대해 모바일 중단점 요구 사항이 정의되어 있습니까? [Gap]"

    명확성:
    *   "'빠른 로딩'이 특정 타이밍 임계값으로 정량화되어 있습니까? [명확성, Spec §NFR-2]"
    *   "'관련 에피소드' 선택 기준이 명시적으로 정의되어 있습니까? [명확성, Spec §FR-5]"
    *   "'눈에 띄는'이 측정 가능한 시각적 속성으로 정의되어 있습니까? [모호성, Spec §FR-4]"

    일관성:
    *   "탐색 요구 사항이 모든 페이지에서 일치합니까? [일관성, Spec §FR-10]"
    *   "카드 구성 요소 요구 사항이 방문 및 세부 정보 페이지 간에 일관됩니까? [일관성]"

    적용 범위:
    *   "제로 상태 시나리오(에피소드 없음)에 대한 요구 사항이 정의되어 있습니까? [적용 범위, 엣지 케이스]"
    *   "동시 사용자 상호 작용 시나리오가 처리됩니까? [적용 범위, Gap]"
    *   "부분 데이터 로딩 실패에 대한 요구 사항이 지정되어 있습니까? [적용 범위, 예외 흐름]"

    측정 가능성:
    *   "시각적 계층 구조 요구 사항이 측정/테스트 가능합니까? [수용 기준, Spec §FR-1]"
    *   "'균형 잡힌 시각적 가중치'를 객관적으로 확인할 수 있습니까? [측정 가능성, Spec §FR-2]"

    **시나리오 분류 및 적용 범위**(요구 사항 품질 초점):
    *   기본, 대체, 예외/오류, 복구, 비기능 시나리오에 대한 요구 사항이 있는지 확인합니다.
    *   각 시나리오 클래스에 대해 "[시나리오 유형] 요구 사항이 완전하고 명확하며 일관됩니까?"라고 질문합니다.
    *   시나리오 클래스가 누락된 경우: "[시나리오 유형] 요구 사항이 의도적으로 제외되었습니까, 아니면 누락되었습니까? [Gap]"
    *   상태 변경이 발생할 때 복원력/롤백을 포함합니다. "마이그레이션 실패에 대한 롤백 요구 사항이 정의되어 있습니까? [Gap]"

    **추적성 요구 사항**:
    *   최소: 항목의 80% 이상이 하나 이상의 추적성 참조를 포함해야 합니다.
    *   각 항목은 사양 섹션 `[Spec §X.Y]`를 참조하거나 마커를 사용해야 합니다: `[Gap]`, `[Ambiguity]`, `[Conflict]`, `[Assumption]`
    *   ID 시스템이 없는 경우: "요구 사항 및 수용 기준 ID 체계가 설정되어 있습니까? [추적성]"

    **문제 표면화 및 해결**(요구 사항 품질 문제):
    요구 사항 자체에 대해 질문합니다.
    *   모호성: "'빠른'이라는 용어가 특정 메트릭으로 정량화되어 있습니까? [모호성, Spec §NFR-1]"
    *   충돌: "§FR-10과 §FR-10a 간에 탐색 요구 사항이 충돌합니까? [충돌]"
    *   가정: "'항상 사용 가능한 팟캐스트 API'라는 가정이 검증되었습니까? [가정]"
    *   종속성: "외부 팟캐스트 API 요구 사항이 문서화되어 있습니까? [종속성, Gap]"
    *   누락된 정의: "'시각적 계층 구조'가 측정 가능한 기준으로 정의되어 있습니까? [Gap]"

    **콘텐츠 통합**:
    *   소프트 캡: 원시 후보 항목이 40개를 초과하는 경우 위험/영향별로 우선순위를 지정합니다.
    *   동일한 요구 사항 측면을 확인하는 거의 중복된 항목을 병합합니다.
    *   영향이 적은 엣지 케이스가 5개 이상인 경우 "요구 사항에서 엣지 케이스 X, Y, Z가 처리됩니까? [적용 범위]"라는 항목 하나를 만듭니다.

    **🚫 절대 금지** - 구현 테스트가 아닌 요구 사항 테스트입니다.
    *   ❌ "확인", "테스트", "확인", "확인" + 구현 동작으로 시작하는 모든 항목
    *   ❌ 코드 실행, 사용자 작업, 시스템 동작에 대한 참조
    *   ❌ "올바르게 표시됨", "제대로 작동함", "예상대로 작동함"
    *   ❌ "클릭", "탐색", "렌더링", "로드", "실행"
    *   ❌ 테스트 케이스, 테스트 계획, QA 절차
    *   ❌ 구현 세부 정보(프레임워크, API, 알고리즘)

    **✅ 필수 패턴** - 요구 사항 품질을 테스트합니다.
    *   ✅ "[시나리오]에 대해 [요구 사항 유형]이 정의/지정/문서화되어 있습니까?"
    *   ✅ "[모호한 용어]가 특정 기준으로 정량화/명확화되어 있습니까?"
    *   ✅ "[섹션 A]와 [섹션 B] 간에 요구 사항이 일관됩니까?"
    *   ✅ "[요구 사항]을 객관적으로 측정/확인할 수 있습니까?"
    *   ✅ "요구 사항에서 [엣지 케이스/시나리오]가 처리됩니까?"
    *   ✅ "사양에서 [누락된 측면]을 정의합니까?"

6.  **구조 참조**: 제목, 메타 섹션, 범주 제목 및 ID 서식에 대해 `.specify/templates/checklist-template.md`의 표준 템플릿에 따라 체크리스트를 생성합니다. 템플릿을 사용할 수 없는 경우 H1 제목, 목적/생성된 메타 라인, CHK001부터 전역적으로 증가하는 ID가 있는 `- [ ] CHK### <요구 사항 항목>` 라인을 포함하는 `##` 범주 섹션을 사용합니다.

7.  **보고서**: 생성된 체크리스트의 전체 경로, 항목 수를 출력하고 각 실행이 새 파일을 생성한다는 것을 사용자에게 상기시킵니다. 요약:
    *   선택한 초점 영역
    *   깊이 수준
    *   행위자/타이밍
    *   명시적으로 사용자가 지정한 필수 항목 통합

**중요**: 각 `/speckit.checklist` 명령 호출은 파일이 이미 존재하지 않는 한 짧고 설명적인 이름을 사용하여 체크리스트 파일을 만듭니다. 이를 통해 다음이 가능합니다.

-   다양한 유형의 여러 체크리스트(예: `ux.md`, `test.md`, `security.md`)
-   체크리스트 목적을 나타내는 간단하고 기억하기 쉬운 파일 이름
-   `checklists/` 폴더에서 쉬운 식별 및 탐색

혼란을 피하려면 설명적인 유형을 사용하고 완료되면 쓸모없는 체크리스트를 정리하십시오.

## 안티-예제: 하지 말아야 할 것

**❌ 잘못됨 - 구현을 테스트하는 것이지 요구 사항을 테스트하는 것이 아닙니다.**

```markdown
- [ ] CHK001 - 방문 페이지에 에피소드 카드 3개가 표시되는지 확인 [Spec §FR-001]
- [ ] CHK002 - 데스크톱에서 호버 상태가 올바르게 작동하는지 테스트 [Spec §FR-003]
- [ ] CHK003 - 로고 클릭이 홈 페이지로 이동하는지 확인 [Spec §FR-010]
- [ ] CHK004 - 관련 에피소드 섹션에 3-5개 항목이 표시되는지 확인 [Spec §FR-005]
```

**✅ 올바름 - 요구 사항 품질을 테스트합니다.**

```markdown
- [ ] CHK001 - 추천 에피소드의 수와 레이아웃이 명시적으로 지정되어 있습니까? [완전성, Spec §FR-001]
- [ ] CHK002 - 모든 대화형 요소에서 호버 상태 요구 사항이 일관되게 정의되어 있습니까? [일관성, Spec §FR-003]
- [ ] CHK003 - 모든 클릭 가능한 브랜드 요소에 대해 탐색 요구 사항이 명확합니까? [명확성, Spec §FR-010]
- [ ] CHK004 - 관련 에피소드에 대한 선택 기준이 문서화되어 있습니까? [Gap, Spec §FR-005]
- [ ] CHK005 - 비동기 에피소드 데이터에 대한 로딩 상태 요구 사항이 정의되어 있습니까? [Gap]
- [ ] CHK006 - "시각적 계층 구조" 요구 사항을 객관적으로 측정할 수 있습니까? [측정 가능성, Spec §FR-001]
```

**주요 차이점:**

-   잘못됨: 시스템이 올바르게 작동하는지 테스트합니다.
-   올바름: 요구 사항이 올바르게 작성되었는지 테스트합니다.
-   잘못됨: 동작 확인
-   올바름: 요구 사항 품질 확인
-   잘못됨: "X를 수행합니까?"
-   올바름: "X가 명확하게 지정되어 있습니까?"
```
