## 사용자 입력

```text
$ARGUMENTS
```

진행하기 전에 사용자 입력을 **반드시** 고려해야 합니다(비어 있지 않은 경우).

## 목표

구현 전에 세 가지 핵심 아티팩트(`spec.md`, `plan.md`, `tasks.md`)에서 불일치, 중복, 모호성 및 불충분하게 지정된 항목을 식별합니다. 이 명령은 `/speckit.tasks`가 완전한 `tasks.md`를 성공적으로 생성한 후에만 실행해야 합니다.

## 운영 제약

**엄격히 읽기 전용**: 파일을 수정하지 **마십시오**. 구조화된 분석 보고서를 출력합니다. 선택적 수정 계획을 제안합니다(사용자는 수동으로 후속 편집 명령을 호출하기 전에 명시적으로 승인해야 함).

**헌법 권한**: 프로젝트 헌법(`.specify/memory/constitution.md`)은 이 분석 범위 내에서 **협상 불가**입니다. 헌법 충돌은 자동으로 치명적이며 원칙의 희석, 재해석 또는 조용한 무시가 아닌 사양, 계획 또는 작업의 조정이 필요합니다. 원칙 자체가 변경되어야 하는 경우 `/speckit.analyze` 외부의 별도 명시적 헌법 업데이트에서 발생해야 합니다.

## 실행 단계

### 1. 분석 컨텍스트 초기화

리포지토리 루트에서 `.specify/scripts/bash/check-prerequisites.sh --json --require-tasks --include-tasks`를 한 번 실행하고 FEATURE_DIR 및 AVAILABLE_DOCS에 대한 JSON을 구문 분석합니다. 절대 경로를 파생합니다.

-   SPEC = FEATURE_DIR/spec.md
-   PLAN = FEATURE_DIR/plan.md
-   TASKS = FEATURE_DIR/tasks.md

필수 파일이 누락된 경우 오류 메시지와 함께 중단합니다(사용자에게 누락된 필수 구성 요소 명령을 실행하도록 지시). "I'm Groot"와 같은 인수에 작은따옴표가 있는 경우 이스케이프 구문을 사용합니다(예: 'I'\''m Groot' 또는 가능한 경우 큰따옴표: "I'm Groot").

### 2. 아티팩트 로드(점진적 공개)

각 아티팩트에서 최소한의 필요한 컨텍스트만 로드합니다.

**spec.md에서:**

-   개요/컨텍스트
-   기능 요구 사항
-   비기능 요구 사항
-   사용자 스토리
-   엣지 케이스(있는 경우)

**plan.md에서:**

-   아키텍처/스택 선택
-   데이터 모델 참조
-   단계
-   기술적 제약

**tasks.md에서:**

-   작업 ID
-   설명
-   단계 그룹화
-   병렬 마커 [P]
-   참조된 파일 경로

**헌법에서:**

-   원칙 유효성 검사를 위해 `.specify/memory/constitution.md` 로드

### 3. 의미론적 모델 구축

내부 표현을 만듭니다(출력에 원시 아티팩트를 포함하지 않음):

-   **요구 사항 인벤토리**: 안정적인 키가 있는 각 기능 + 비기능 요구 사항(명령문 구문을 기반으로 슬러그 파생, 예: "사용자는 파일을 업로드할 수 있습니다" → `user-can-upload-file`)
-   **사용자 스토리/작업 인벤토리**: 수용 기준이 있는 개별 사용자 작업
-   **작업 적용 범위 매핑**: 각 작업을 하나 이상의 요구 사항 또는 스토리에 매핑(키워드/명시적 참조 패턴(ID 또는 핵심 구문 등)에 의한 추론)
-   **헌법 규칙 세트**: 원칙 이름 및 MUST/SHOULD 규범적 진술 추출

### 4. 탐지 패스(토큰 효율적인 분석)

고신호 결과에 집중합니다. 총 50개 결과로 제한하고 나머지는 오버플로 요약에 집계합니다.

#### A. 중복 탐지

-   거의 중복되는 요구 사항 식별
-   통합을 위해 품질이 낮은 표현 표시

#### B. 모호성 탐지

-   측정 가능한 기준이 없는 모호한 형용사(빠른, 확장 가능한, 안전한, 직관적인, 견고한) 플래그 지정
-   미해결 자리 표시자(TODO, TKTK, ???, `<placeholder>` 등) 플래그 지정

#### C. 불충분한 사양

-   동사는 있지만 목적어 또는 측정 가능한 결과가 없는 요구 사항
-   수용 기준 정렬이 누락된 사용자 스토리
-   사양/계획에 정의되지 않은 파일 또는 구성 요소를 참조하는 작업

#### D. 헌법 정렬

-   MUST 원칙과 충돌하는 모든 요구 사항 또는 계획 요소
-   헌법에서 위임된 섹션 또는 품질 게이트 누락

#### E. 적용 범위 격차

-   연관된 작업이 없는 요구 사항
-   매핑된 요구 사항/스토리가 없는 작업
-   작업에 반영되지 않은 비기능적 요구 사항(예: 성능, 보안)

#### F. 불일치

-   용어 드리프트(파일 전체에서 동일한 개념이 다르게 명명됨)
-   계획에서 참조되었지만 사양에 없는 데이터 엔터티(또는 그 반대)
-   작업 순서 모순(예: 종속성 참고 없이 기본 설정 작업 전에 통합 작업)
-   충돌하는 요구 사항(예: 하나는 Next.js를 요구하고 다른 하나는 Vue를 지정함)

### 5. 심각도 할당

이 휴리스틱을 사용하여 결과 우선순위를 지정합니다.

-   **치명적**: 헌법 MUST 위반, 핵심 사양 아티팩트 누락 또는 기본 기능을 차단하는 적용 범위가 없는 요구 사항
-   **높음**: 중복되거나 충돌하는 요구 사항, 모호한 보안/성능 속성, 테스트할 수 없는 수용 기준
-   **중간**: 용어 드리프트, 비기능적 작업 적용 범위 누락, 불충분하게 지정된 엣지 케이스
-   **낮음**: 스타일/표현 개선, 실행 순서에 영향을 미치지 않는 사소한 중복

### 6. 간결한 분석 보고서 생성

다음 구조로 마크다운 보고서(파일 쓰기 없음)를 출력합니다.

### 7. 다음 작업 제공

보고서 끝에 간결한 다음 작업 블록을 출력합니다.

-   치명적인 문제가 있는 경우: `/speckit.implement` 전에 해결 권장
-   낮음/중간만 있는 경우: 사용자는 진행할 수 있지만 개선 제안 제공
-   명시적인 명령 제안 제공: 예: "개선을 통해 /speckit.specify 실행", "아키텍처 조정을 위해 /speckit.plan 실행", "'성능 메트릭'에 대한 적용 범위를 추가하기 위해 tasks.md 수동 편집"

### 8. 수정 제안

사용자에게 "상위 N개 문제에 대한 구체적인 수정 편집을 제안하시겠습니까?"라고 묻습니다. (자동으로 적용하지 않음)

## 운영 원칙

### 컨텍스트 효율성

-   **최소한의 고신호 토큰**: 전체 문서가 아닌 실행 가능한 결과에 집중
-   **점진적 공개**: 아티팩트를 점진적으로 로드하고 분석에 모든 콘텐츠를 덤프하지 않음
-   **토큰 효율적인 출력**: 결과 테이블을 50행으로 제한하고 오버플로 요약
-   **결정론적 결과**: 변경 없이 다시 실행하면 일관된 ID 및 개수 생성

### 분석 지침

-   **파일을 절대 수정하지 마십시오**(읽기 전용 분석임)
-   **누락된 섹션을 절대 추측하지 마십시오**(없는 경우 정확하게 보고)
-   **헌법 위반 우선순위 지정**(항상 치명적임)
-   **전체 규칙보다 예제 사용**(일반적인 패턴이 아닌 특정 인스턴스 인용)
-   **문제 없음 정상적으로 보고**(적용 범위 통계와 함께 성공 보고서 내보내기)
