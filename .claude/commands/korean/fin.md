다음 작업을 순서대로 실행하세요:

## 1. 📄 살아있는 문서 동기화 (/ms.up-docs)

**신규**: 코드 변경사항을 살아있는 문서에 반영합니다.

### 실행

```bash
/ms.up-docs --docs=dev
```

**동작**:
- **Git 변경사항 분석**: `git diff --cached` (스테이징된 파일만)
- **dev_daily.md 자동 업데이트**: Git diff 요약을 타임스탬프와 함께 추가
- **TAG 체인 검증**: @SPEC → @TEST → @CODE 무결성 확인
- **API 문서 동기화** (스테이징된 변경사항 포함 시): `docs/api/{TAG}.md` 생성/업데이트

**출력 예시**:
```
✅ 문서 동기화 완료

📦 업데이트된 파일:
- docs/dev_daily.md (추가됨)
- docs/api/AUTH-001.md (업데이트됨)

📋 TAG 무결성: 95.0% (24개 중 23개 체인 완료)

⏱️ 소요 시간: 3.2초
```

**오류 처리**:
- **스테이징된 변경사항 없음**: ⚠️ 경고 출력 후 다음 단계 진행 (dev_daily.md 수동 업데이트로 대체)
- **TAG 무결성 낮음** (<80%): ⚠️ 경고 출력하지만 워크플로우 계속 진행

---

## 2. 📝 일일 작업 로그 검토 (대체)

**참고**: 1단계의 `/ms.up-docs`가 dev_daily.md를 이미 업데이트했습니다.
이 단계는 `/ms.up-docs` 실패 시 대체용으로만 사용됩니다.

현재 세션의 작업 내용을 분석하여 `docs/dev_daily.md`를 검토하세요:

### 분석 항목
- 변경된 파일 목록 확인 (git status 또는 대화 내용 기반)
- 주요 작업 내용 파악
- 추가/수정/삭제된 기능

### 업데이트 규칙
1. **오늘 날짜 섹션 확인**:
   - 오늘 날짜(YYYY-MM-DD)의 섹션이 있으면 → 해당 섹션의 완료 항목에 추가
   - 없으면 → 파일 최상단에 새 섹션 추가

2. **작성 형식** (기존 양식 유지):
   ```markdown
   # 🗓 YYYY-MM-DD (요일)

   ## 📌 집중
   - 주요 작업 테마 (1줄 요약)

   ## ✅ 완료
   - 완료한 작업 내역 (간략하게, 3-5개 항목)
   ```

3. **작성 스타일**:
   - **간결하게**: 각 항목은 1줄로
   - **구체적으로**: "코드 수정" (X) → "tech_algo.py에 MACD 신호 추가" (O)
   - **기존 항목 유지**: 오늘 날짜 섹션에 이미 작성된 내용이 있으면 완료 항목 아래에 추가 (덮어쓰기 X)

### 예시
```markdown
# 🗓 2025-10-01 (화)

## 📌 집중
- AGENTS.md 템플릿 구조 개선 및 Makefile 통합

## ✅ 완료
- templates 디렉토리 생성 및 AGENTS.md 3개 파일 작성 완료
- Markdown 서식 적용 (헤더, 코드블록, 목록)
- 루트 Makefile과 .devcontainer/Makefile 역할 분리
- /fin, /finq 슬래시 커맨드 추가
```

## 3. 🚀 CI 확인 실행

**에이전트 위임**: 이 단계에서는 커밋하기 전에 TRUST 유효성 검사를 수행하기 위해 **quality-gate** 에이전트(Haiku 모델)를 사용합니다.
- 테스트 커버리지 ≥85% (필수)
- 모든 테스트 통과
- 타입 오류 없음
- 린트 경고 없음
- TAG 무결성 유효성 검사

다음 명령어를 실행하여 코드 품질을 검증합니다:

```bash
make ci
```

**CI 포함 내용**:
- `black --check .` (코드 서식)
- `isort --check-only .` (import 정렬)
- `ruff check .` (린트)
- `mypy src/` (타입 확인)
- `pytest -v --cov=src` (테스트 ≥85% 커버리지)

**결과 처리**:
- ✅ **통과**: 다음 단계 진행
- ❌ **실패**: 구체적인 오류 메시지 출력 후 **중단** (커밋하지 않음)

---

## 4. 💾 Git 커밋 및 푸시

### 사전 커밋 훅 처리 전략

**문제**: 사전 커밋 훅이나 IDE의 자동 서식이 커밋 후 파일을 수정하여 git 상태가 더러워짐

**해결 방법**: 커밋 전에 사전 커밋을 먼저 실행하여 서식 완료

```bash
# 1. 초기 git add
git add .

# 2. 사전 커밋 훅을 실행하여 파일 서식 지정 (.pre-commit-config.yaml이 있는 경우)
if [ -f .pre-commit-config.yaml ]; then
  pre-commit run --all-files || true
  # 훅이 파일을 수정할 수 있으므로 다시 추가
  git add .
fi

# 3. git commit (커밋 메시지 생성)
git commit -m "생성한_커밋_메시지"

# 4. git push
git push
```

**작동 원리**:
- 사전 커밋 훅(ruff-format 등)이 실행되어 파일을 수정합니다.
- 수정된 파일은 두 번째 `git add .`로 다시 스테이징됩니다.
- 커밋은 모든 서식 변경 사항을 캡처합니다.
- 커밋 후 더러운 상태가 없습니다.

### 커밋 메시지 생성 규칙
변경 사항을 분석하여 **의미 있는 커밋 메시지** 작성:

**형식**:
```
타입(범위): 제목

- 상세 내용 1
- 상세 내용 2
```

**타입**:
- `feat`: 새 기능
- `fix`: 버그 수정
- `refactor`: 리팩토링
- `docs`: 문서 수정
- `test`: 테스트 추가
- `chore`: 빌드, 설정 변경

**예시**:
```
feat(전략): MACD 기반 기술적 분석 추가

- tech_algo.py에 MACD 신호 생성 로직 구현
- 백테스트 테스트 케이스 추가
```

### 오류 처리
- **커밋할 내용이 없을 때**: `⚠️ 커밋할 내용 없음` 메시지 출력 후 계속 진행
- **푸시 실패 시**: `❌ 푸시 실패` 메시지 출력 (오류 내용 포함)

---

## 5. ✅ 완료 메시지

모든 단계 완료 후 출력:

```
✅ /fin 완료!

📄 살아있는 문서 동기화 완료 (/ms.up-docs)
📝 dev_daily.md 업데이트 완료
🚀 CI 확인 통과
💾 커밋: [생성한 메시지]
🚀 푸시 완료 (또는 실패 시 오류 메시지)
```

---

## ⚠️ 주의사항

1. **/ms.up-docs 통합**:
   - **신규**: 1단계에서 `/ms.up-docs --docs=dev` 자동 실행
   - dev_daily.md가 자동으로 업데이트됨 (Git diff 기반)
   - TAG 체인 무결성 자동 검증
   - 스테이징된 변경사항 없으면 경고 후 계속 진행 (수동 업데이트로 대체)

2. **dev_daily.md 업데이트 시** (대체만 해당):
   - 기존 내용을 절대 삭제하지 마세요
   - 오늘 날짜 섹션에 내용 추가만
   - 집중은 오늘 날짜 섹션이 처음 생성될 때만 작성

3. **CI 실패 시**:
   - 즉시 중단
   - 어떤 확인이 실패했는지 명확히 출력
   - 커밋/푸시 하지 않음

4. **Git 동작**:
   - Makefile의 finish 타겟과 동일하게 처리
   - 커밋 실패 시에도 오류 메시지만 출력하고 계속 진행
   - 푸시 실패 시에도 오류 메시지만 출력하고 워크플로우 완료

5. **워크플로우 순서 (업데이트됨)**:
   ```
   /ms.up-docs --docs=dev  # 1단계: 살아있는 문서 동기화
   ↓
   (dev_daily.md 검토)     # 2단계: 대체만 해당
   ↓
   make ci                 # 3단계: CI 확인
   ↓
   git commit && push      # 4단계: 커밋 및 푸시
   ```
