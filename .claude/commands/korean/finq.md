## 1. 📄 살아있는 문서 동기화(/ms.up-docs)

**신규**: 코드 변경 사항을 살아있는 문서에 빠르게 반영합니다.

### 실행

```bash
/ms.up-docs --docs=dev
```

**동작**:
- **Git 변경 사항 분석**: `git diff --cached` (스테이징된 파일만)
- **dev_daily.md 자동 업데이트**: Git diff 요약을 타임스탬프와 함께 추가
- **TAG 체인 유효성 검사**: @SPEC → @TEST → @CODE 무결성 확인(빠른 스캔)
- **API 문서 동기화**(스테이징된 변경 사항 포함 시): `docs/api/{TAG}.md` 생성/업데이트

**빠른 모드 최적화**:
- 스테이징된 변경 사항만 처리(전체 스캔 없음)
- TAG 유효성 검사는 기본만 수행(상세 유효성 검사 생략)
- API 문서는 변경된 파일만 업데이트

**출력 예시**:
```
✅ 문서 동기화 완료(빠른 모드)

📦 업데이트된 파일:
- docs/dev_daily.md (추가됨)

📋 TAG 무결성: 95.0%(빠른 스캔)

⏱️ 소요 시간: 1.8초
```

**오류 처리**:
- **스테이징된 변경 사항 없음**: ⚠️ 경고 출력 후 다음 단계 진행(dev_daily.md 수동 업데이트로 대체)
- **TAG 무결성 낮음**(<80%): ⚠️ 경고를 출력하지만 워크플로 계속 진행(나중에 /fin으로 유효성 검사)

## 2. 📝 일일 작업 로그 검토(대체)

**참고**: 1단계의 `/ms.up-docs`가 dev_daily.md를 이미 업데이트했습니다.
이 단계는 `/ms.up-docs` 실패 시 대체용으로만 사용됩니다.

현재 세션의 작업 내용을 분석하여 `docs/dev_daily.md`를 검토하세요:

### 분석 항목
- 변경된 파일 목록 확인(git status 또는 대화 내용 기반)
- 주요 작업 내용 파악
- 추가/수정/삭제된 기능

### 업데이트 규칙
1. **오늘 날짜 섹션 확인**:
   - 오늘 날짜(YYYY-MM-DD)의 섹션이 있으면 → 해당 섹션의 완료 항목에 추가
   - 없으면 → 파일 최상단에 새 섹션 추가

2. **작성 형식**(기존 양식 유지):
   ```markdown
   # 🗓 YYYY-MM-DD (요일)

   ## 📌 집중
   - 주요 작업 테마(1줄 요약)

   ## ✅ 완료
   - 완료한 작업 내역(간략하게, 3-5개 항목)
   ```

3. **작성 스타일**:
   - **간결하게**: 각 항목은 1줄로
   - **구체적으로**: "코드 수정"(X) → "tech_algo.py에 MACD 신호 추가"(O)
   - **기존 항목 유지**: 오늘 날짜 섹션에 이미 작성된 내용이 있으면 완료 항목 아래에 추가(덮어쓰기 X)

### 예시
```markdown
# 🗓 2025-10-01 (화)

## 📌 집중
- AGENTS.md 템플릿 구조 개선 및 Makefile 통합

## ✅ 완료
- templates 디렉토리 생성 및 AGENTS.md 3개 파일 작성 완료
- Markdown 서식 적용(헤더, 코드 블록, 목록)
- 루트 Makefile과 .devcontainer/Makefile 역할 분리
- /fin, /finq 슬래시 커맨드 추가
```

## 2.5. 📄 README.md 자동 최신화(선택 사항)

### 분석 항목

변경된 파일 중 주요 기능 변경 사항이 있는지 확인:

**주요 변경 사항 감지 대상**:
- `.claude/agents/*` - 에이전트 구조 변경
- `.claude/commands/*` - 워크플로 변경
- `.devcontainer/Makefile` - 개발 명령어 변경
- `pyproject.toml` - 의존성/기술 스택 변경
- `.mcp.json` - MCP 서버 설정 변경

### 업데이트 판단 기준

다음 변경 사항이 있을 경우 **README.md 업데이트 필수**:
- ✅ 새로운 주요 기능 추가(에이전트, 명령어, 워크플로)
- ✅ 프로젝트 구조 변경(디렉토리, 핵심 파일)
- ✅ 설치/실행 방법 변경(Makefile 대상, 환경 변수)
- ✅ 기술 스택 변경(의존성 관리 방식, MCP 서버)

**업데이트 건너뛰기**:
- ❌ 문서만 수정(`docs/*.md`)
- ❌ 설정 파일 미세 조정
- ❌ 버그 수정(기능 변경 없음)

### 업데이트 원칙(중요)

**❌ 절대 금지 - 변경 이력 추적**:
```markdown
❌ "변경됨", "추가됨", "업데이트됨" 같은 메타 정보
❌ 변경 날짜나 버전 정보
❌ 과거 상태 언급("이전에는 X였으나 이제 Y")
❌ "(2025-10-21 추가)", "(최근 변경)" 같은 타임스탬프
```

**✅ 필수 사항 - 현재 상태만 기술**:
```markdown
✅ 프로젝트를 처음 보는 사람 관점으로 작성
✅ 지금 당장 사용 가능한 기능/방법만 나열
✅ 현재 상태를 명확하고 간결하게 설명
✅ 시제: 현재형만 사용("제공합니다", "사용합니다", "지원합니다")
```

### 잘못된 예시 vs 올바른 예시

**❌ 잘못된 업데이트(변경 이력 추적)**:
```markdown
## 기능
- 사용자 인증(추가됨: 2025-10-20)
- 에이전트 시스템(변경됨: 2025-10-21)
  - 이전에는 Claude만 사용했으나 이제 Gemini/Codex 분담 구조로 개선
- MCP 서버 지원(최근 추가)

## 변경 사항
- 2025-10-21: 에이전트 시스템이 Gemini/Codex를 사용하도록 업데이트됨
- 2025-10-20: MCP 서버 지원 추가
```

**✅ 올바른 업데이트(현재 상태만)**:
```markdown
## 기능
- 사용자 인증
- 다중 AI 에이전트 시스템
  - Gemini: 빠른 코드베이스 탐색 및 라이브러리 조사
  - Claude: 통합 설계 및 오케스트레이션
  - Codex: 심층 코드 분석 및 유효성 검사
- MCP 서버 통합(Context7, CLI-bridge)

## 빠른 시작
```bash
make sptcc  # 복제 + 컨테이너 + Claude 코드
```

## 프로젝트 구조
```
.claude/
  agents/     # 6개의 전문 에이전트
  commands/   # 워크플로 자동화
```
```

### 구현 로직

```python
# 1. 변경 사항 분석
changed_files = git_status()

# 2. 주요 변경 사항 감지
major_change_patterns = [
    ".claude/agents/",
    ".claude/commands/",
    ".devcontainer/Makefile",
    "pyproject.toml",
    ".mcp.json"
]

has_major_changes = any(
    pattern in file for file in changed_files
    for pattern in major_change_patterns
)

# 3. README.md 업데이트(주요 변경 사항이 있는 경우에만)
if has_major_changes:
    # 현재 README.md 읽기
    readme_content = read_file("README.md")

    # 현재 상태를 반영하도록 섹션 업데이트
    # "변경됨", "추가됨", "업데이트됨" 언급 없음
    # 타임스탬프 또는 버전 정보 없음
    # 현재 존재하는 것만 설명

    updated_readme = update_to_current_state(
        readme_content,
        changed_files
    )

    write_file("README.md", updated_readme)

    print("✅ README.md가 현재 상태를 반영하도록 업데이트되었습니다")
else:
    print("⏭️  주요 변경 사항 없음 - README.md 업데이트 건너뜀")
```

### 업데이트 체크리스트

업데이트 전 확인:
- [ ] 변경 이력 언급 제거("추가됨", "변경됨" 등)
- [ ] 날짜/버전 정보 제거
- [ ] 과거 상태 비교 제거("이전에는...")
- [ ] 현재형 시제 사용("제공합니다", "사용합니다")
- [ ] 처음 보는 사람도 이해 가능한 설명
- [ ] 실제 사용 가능한 명령어/방법만 기재

## 3. 💾 Git 커밋 및 푸시(CI 확인 생략)

### 사전 커밋 훅 처리 전략

**문제**: 사전 커밋 훅이나 IDE의 자동 서식이 커밋 후 파일을 수정하여 git 상태가 더러워짐

**해결 방법**: 커밋 전에 사전 커밋을 먼저 실행하여 서식 완료

```bash
# 1. 초기 git add
git add .

# 2. 사전 커밋 훅을 실행하여 파일 서식 지정(.pre-commit-config.yaml이 있는 경우)
if [ -f .pre-commit-config.yaml ]; then
  pre-commit run --all-files || true
  # 훅이 파일을 수정할 수 있으므로 다시 추가
  git add .
fi

# 3. git commit (커밋 메시지 생성)
git commit -m "생성한_커밋_메시지"

# 4. git push
git push
```

**작동 원리**:
- 사전 커밋 훅(ruff-format 등)이 실행되어 파일을 수정합니다.
- 수정된 파일은 두 번째 `git add .`로 다시 스테이징됩니다.
- 커밋은 모든 서식 변경 사항을 캡처합니다.
- 커밋 후 더러운 상태가 없습니다.

### 커밋 메시지 생성 규칙
변경 사항을 분석하여 **의미 있는 커밋 메시지** 작성:

**형식**:
```
타입(범위): 제목

- 상세 내용 1
- 상세 내용 2
```

**타입**:
- `feat`: 새 기능
- `fix`: 버그 수정
- `refactor`: 리팩토링
- `docs`: 문서 수정
- `test`: 테스트 추가
- `chore`: 빌드, 설정 변경
- `wip`: 작업 중

**예시**:
```
wip(전략): MACD 신호 생성 로직 작업 중

- 기본 구조 구현
- 테스트 작성 필요
```

### 오류 처리
- **커밋할 내용이 없을 때**: `⚠️ 커밋할 내용 없음` 메시지 출력 후 계속 진행
- **푸시 실패 시**: `❌ 푸시 실패` 메시지 출력(오류 내용 포함)

## 4. ✅ 완료 메시지

모든 단계 완료 후 출력:

```
✅ /finq 완료! (빠른 모드 - CI 생략)

📄 살아있는 문서 동기화 완료(/ms.up-docs)
📝 dev_daily.md 업데이트 완료
💾 커밋: [생성한 메시지]
🚀 푸시 완료(또는 실패 시 오류 메시지)

⚠️ CI 확인을 생략했습니다. 나중에 'make ci' 또는 '/fin'으로 확인하세요.
```

## ⚠️ 주의사항

1. **/ms.up-docs 통합(빠른 모드)**:
   - **신규**: 1단계에서 `/ms.up-docs --docs=dev` 자동 실행
   - dev_daily.md가 자동으로 업데이트됨(Git diff 기반)
   - TAG 체인 무결성 빠른 유효성 검사(상세 유효성 검사는 /fin에서 수행)
   - 스테이징된 변경 사항 없으면 경고 후 계속 진행(수동 업데이트로 대체)
   - **성능**: 스테이징된 변경 사항만 처리하여 속도 최적화(약 2초)

2. **dev_daily.md 업데이트 시**(대체만 해당):
   - 기존 내용을 절대 삭제하지 마세요
   - 오늘 날짜 섹션에 내용 추가만
   - 집중은 오늘 날짜 섹션이 처음 생성될 때만 작성

3. **CI 생략됨**:
   - 이 명령어는 빠른 커밋을 위해 CI 확인을 건너뜁니다.
   - 코드 품질 유효성 검사는 나중에 `/fin` 또는 `make ci`로 수행하세요.
   - 작업 중(WIP) 커밋에 적합합니다.

4. **Git 동작**:
   - Makefile의 finish 대상과 동일하게 처리(CI 제외)
   - 커밋 실패 시에도 오류 메시지만 출력하고 계속 진행
   - 푸시 실패 시에도 오류 메시지만 출력하고 워크플로 완료

5. **사용 시나리오**:
   - 작업 중간 백업 목적
   - 빠르게 원격에 올려야 할 때
   - 문서만 수정했을 때
   - 실험적인 코드 커밋

6. **워크플로 순서(업데이트됨)**:
   ```
   /ms.up-docs --docs=dev  # 1단계: 살아있는 문서 동기화(빠름)
   ↓
   (dev_daily.md 검토)     # 2단계: 대체만 해당
   ↓
   (README.md 업데이트)    # 2.5단계: 선택 사항
   ↓
   git commit && push      # 3단계: 커밋 및 푸시(CI 생략)
   ```

7. **/fin vs /finq 차이점**:
   - `/fin`: **전체** - /ms.up-docs → CI → 커밋 → 푸시
   - `/finq`: **빠름** - /ms.up-docs(스테이징된 항목만) → 커밋 → 푸시(CI 생략)
   - TAG 유효성 검사: /fin은 상세, /finq는 빠른 스캔
   - API 문서: /fin은 전체 동기화 가능, /finq는 스테이징된 변경 사항만
