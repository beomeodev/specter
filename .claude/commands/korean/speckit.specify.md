## 사용자 입력

```text
$ARGUMENTS
```

진행하기 전에 사용자 입력을 **반드시** 고려해야 합니다(비어 있지 않은 경우).

## 개요

사용자가 트리거 메시지에서 `/speckit.specify` 뒤에 입력한 텍스트가 기능 설명입니다. `$ARGUMENTS`가 문자 그대로 아래에 표시되더라도 이 대화에서 항상 사용할 수 있다고 가정합니다. 사용자가 빈 명령을 제공하지 않는 한 반복하도록 요청하지 마십시오.

해당 기능 설명이 주어지면 다음을 수행합니다.

1.  **브랜치에 대한 간결한 짧은 이름 생성**(2-4단어):
    *   기능 설명을 분석하고 가장 의미 있는 키워드를 추출합니다.
    *   기능의 본질을 포착하는 2-4단어의 짧은 이름을 만듭니다.
    *   가능하면 동사-명사 형식을 사용합니다(예: "add-user-auth", "fix-payment-bug").
    *   기술 용어 및 약어(OAuth2, API, JWT 등)를 보존합니다.
    *   기능을 한눈에 이해할 수 있을 만큼 간결하지만 설명적으로 유지합니다.
    *   예시:
        *   "사용자 인증을 추가하고 싶습니다" → "user-auth"
        *   "API에 대한 OAuth2 통합 구현" → "oauth2-api-integration"
        *   "분석용 대시보드 만들기" → "analytics-dashboard"
        *   "결제 처리 시간 초과 버그 수정" → "fix-payment-timeout"

2.  **새 브랜치를 만들기 전에 기존 브랜치 확인**:

    a. 먼저 모든 원격 브랜치를 가져와 최신 정보가 있는지 확인합니다.
       ```bash
       git fetch --all --prune
       ```

    b. 짧은 이름에 대한 모든 소스에서 가장 높은 기능 번호를 찾습니다.
       - 원격 브랜치: `git ls-remote --heads origin | grep -E 'refs/heads/[0-9]+-<short-name>$'`
       - 로컬 브랜치: `git branch | grep -E '^[* ]*[0-9]+-<short-name>$'`
       - 사양 디렉토리: `specs/[0-9]+-<short-name>`과 일치하는 디렉토리 확인

    c. 사용 가능한 다음 번호를 결정합니다.
       - 세 소스 모두에서 모든 번호를 추출합니다.
       - 가장 높은 번호 N을 찾습니다.
       - 새 브랜치 번호에 N+1을 사용합니다.

    d. 계산된 번호와 짧은 이름으로 `.specify/scripts/bash/create-new-feature.sh --json "$ARGUMENTS"` 스크립트를 실행합니다.
       - 기능 설명과 함께 `--number N+1` 및 `--short-name "your-short-name"`을 전달합니다.
       - Bash 예시: `.specify/scripts/bash/create-new-feature.sh --json "$ARGUMENTS" --json --number 5 --short-name "user-auth" "사용자 인증 추가"`
       - PowerShell 예시: `.specify/scripts/bash/create-new-feature.sh --json "$ARGUMENTS" -Json -Number 5 -ShortName "user-auth" "사용자 인증 추가"`

    **중요**:
    - 세 소스(원격 브랜치, 로컬 브랜치, 사양 디렉토리)를 모두 확인하여 가장 높은 번호를 찾습니다.
    - 정확한 짧은 이름 패턴과 일치하는 브랜치/디렉토리만 일치시킵니다.
    - 이 짧은 이름으로 기존 브랜치/디렉토리를 찾을 수 없는 경우 1번부터 시작합니다.
    - 기능당 이 스크립트를 한 번만 실행해야 합니다.
    - JSON은 터미널에 출력으로 제공됩니다. 찾고 있는 실제 콘텐츠를 얻으려면 항상 참조하십시오.
    - JSON 출력에는 BRANCH_NAME 및 SPEC_FILE 경로가 포함됩니다.
    - "I'm Groot"와 같은 인수에 작은따옴표가 있는 경우 이스케이프 구문을 사용합니다(예: 'I'\''m Groot' 또는 가능한 경우 큰따옴표: "I'm Groot").

3.  `.specify/templates/spec-template.md`를 로드하여 필수 섹션을 이해합니다.

4.  다음 실행 흐름을 따릅니다.

    1.  입력에서 사용자 설명 구문 분석
        비어 있는 경우: 오류 "기능 설명이 제공되지 않았습니다"
    2.  설명에서 핵심 개념 추출
        식별: 행위자, 행동, 데이터, 제약 조건
    3.  불분명한 측면의 경우:
        *   컨텍스트 및 업계 표준을 기반으로 정보에 입각한 추측을 합니다.
        *   다음과 같은 경우에만 [명확화 필요: 특정 질문]으로 표시합니다.
            *   선택이 기능 범위나 사용자 경험에 큰 영향을 미칩니다.
            *   서로 다른 의미를 가진 여러 합리적인 해석이 존재합니다.
            *   합리적인 기본값이 없습니다.
        *   **제한: 총 3개의 [명확화 필요] 마커**
        *   영향별로 명확화 우선순위 지정: 범위 > 보안/개인 정보 > 사용자 경험 > 기술적 세부 정보
    4.  사용자 시나리오 및 테스트 섹션 채우기
        명확한 사용자 흐름이 없는 경우: 오류 "사용자 시나리오를 결정할 수 없습니다"
    5.  기능 요구 사항 생성
        각 요구 사항은 테스트 가능해야 합니다.
        지정되지 않은 세부 정보에 대해 합리적인 기본값을 사용합니다(가정 섹션에 가정 문서화).
    6.  성공 기준 정의
        측정 가능하고 기술에 구애받지 않는 결과 만들기
        정량적 메트릭(시간, 성능, 볼륨)과 정성적 측정(사용자 만족도, 작업 완료)을 모두 포함합니다.
        각 기준은 구현 세부 정보 없이 확인할 수 있어야 합니다.
    7.  주요 엔터티 식별(데이터 관련 시)
    8.  반환: 성공(계획 준비 완료된 사양)

5.  템플릿 구조를 사용하여 사양을 SPEC_FILE에 작성하고, 섹션 순서와 제목을 유지하면서 자리 표시자를 기능 설명(인수)에서 파생된 구체적인 세부 정보로 바꿉니다.

6.  **사양 품질 검증**: 초기 사양을 작성한 후 품질 기준에 대해 검증합니다.

    a. **사양 품질 체크리스트 만들기**: `FEATURE_DIR/checklists/requirements.md`에 다음 검증 항목이 포함된 체크리스트 템플릿 구조를 사용하여 체크리스트 파일을 생성합니다.

       ```markdown
       # 사양 품질 체크리스트: [기능 이름]

       **목적**: 계획 진행 전에 사양 완전성 및 품질 검증
       **생성일**: [날짜]
       **기능**: [spec.md 링크]

       ## 콘텐츠 품질

       - [ ] 구현 세부 정보 없음(언어, 프레임워크, API)
       - [ ] 사용자 가치 및 비즈니스 요구에 중점
       - [ ] 비기술적 이해관계자를 위해 작성됨
       - [ ] 모든 필수 섹션 완료

       ## 요구 사항 완전성

       - [ ] [명확화 필요] 마커 남지 않음
       - [ ] 요구 사항은 테스트 가능하고 명확함
       - [ ] 성공 기준은 측정 가능함
       - [ ] 성공 기준은 기술에 구애받지 않음(구현 세부 정보 없음)
       - [ ] 모든 수용 시나리오 정의됨
       - [ ] 엣지 케이스 식별됨
       - [ ] 범위가 명확하게 한정됨
       - [ ] 종속성 및 가정 식별됨

       ## 기능 준비 상태

       - [ ] 모든 기능 요구 사항에 명확한 수용 기준이 있음
       - [ ] 사용자 시나리오가 기본 흐름을 다룸
       - [ ] 기능이 성공 기준에 정의된 측정 가능한 결과를 충족함
       - [ ] 구현 세부 정보가 사양에 유출되지 않음

       ## 참고

       - 불완전으로 표시된 항목은 `/speckit.clarify` 또는 `/speckit.plan` 전에 사양 업데이트가 필요함
       ```

    b. **검증 실행**: 각 체크리스트 항목에 대해 사양을 검토합니다.
       - 각 항목에 대해 통과 또는 실패 여부를 결정합니다.
       - 발견된 특정 문제를 문서화합니다(관련 사양 섹션 인용).

    c. **검증 결과 처리**:

       - **모든 항목이 통과하는 경우**: 체크리스트를 완료로 표시하고 6단계로 진행합니다.

       - **항목이 실패하는 경우([명확화 필요] 제외)**:
         1.  실패한 항목과 특정 문제를 나열합니다.
         2.  각 문제를 해결하도록 사양을 업데이트합니다.
         3.  모든 항목이 통과할 때까지 검증을 다시 실행합니다(최대 3회 반복).
         4.  3회 반복 후에도 여전히 실패하는 경우 남은 문제를 체크리스트 노트에 문서화하고 사용자에게 경고합니다.

       - **[명확화 필요] 마커가 남은 경우**:
         1.  사양에서 모든 [명확화 필요: ...] 마커를 추출합니다.
         2.  **제한 확인**: 3개 이상의 마커가 있는 경우 가장 중요한 3개(범위/보안/UX 영향 기준)만 유지하고 나머지는 정보에 입각한 추측을 합니다.
         3.  필요한 각 명확화(최대 3개)에 대해 다음 형식으로 사용자에게 옵션을 제시합니다.

            ```markdown
            ## 질문 [N]: [주제]

            **컨텍스트**: [관련 사양 섹션 인용]

            **알아야 할 사항**: [명확화 필요 마커의 특정 질문]

            **제안된 답변**:

            | 옵션 | 답변 | 영향 |
            |---|---|---|
            | A | [첫 번째 제안된 답변] | [이것이 기능에 미치는 영향] |
            | B | [두 번째 제안된 답변] | [이것이 기능에 미치는 영향] |
            | C | [세 번째 제안된 답변] | [이것이 기능에 미치는 영향] |
            | 사용자 지정 | 직접 답변 제공 | [사용자 지정 입력 제공 방법 설명] |

            **선택**: _[사용자 응답 대기]_
            ```

         4.  **중요 - 테이블 서식**: 마크다운 테이블이 올바르게 서식되었는지 확인합니다.
             *   파이프가 정렬된 일관된 간격을 사용합니다.
             *   각 셀 주위에 공백이 있어야 합니다: `| 내용 |`이 아니라 `|내용|`
             *   헤더 구분선에는 최소 3개의 대시가 있어야 합니다: `|---|`
             *   마크다운 미리보기에서 테이블이 올바르게 렌더링되는지 테스트합니다.
         5.  질문을 순차적으로 번호 매기기(Q1, Q2, Q3 - 최대 3개).
         6.  응답을 기다리기 전에 모든 질문을 함께 제시합니다.
         7.  사용자가 모든 질문에 대한 선택으로 응답할 때까지 기다립니다(예: "Q1: A, Q2: 사용자 지정 - [세부 정보], Q3: B").
         8.  각 [명확화 필요] 마커를 사용자가 선택하거나 제공한 답변으로 바꾸어 사양을 업데이트합니다.
         9.  모든 명확화가 해결된 후 검증을 다시 실행합니다.

    d. **체크리스트 업데이트**: 각 검증 반복 후 현재 통과/실패 상태로 체크리스트 파일을 업데이트합니다.

7.  브랜치 이름, 사양 파일 경로, 체크리스트 결과 및 다음 단계(`/speckit.clarify` 또는 `/speckit.plan`)에 대한 준비 상태와 함께 완료를 보고합니다.

**참고:** 스크립트는 새 브랜치를 만들고 체크아웃하며 쓰기 전에 사양 파일을 초기화합니다.

## 일반 지침

## 빠른 지침

-   사용자가 **무엇을** 필요로 하고 **왜** 필요한지에 집중합니다.
-   구현 방법(기술 스택, API, 코드 구조 없음)을 피합니다.
-   개발자가 아닌 비즈니스 이해관계자를 위해 작성되었습니다.
-   사양에 포함된 체크리스트를 만들지 마십시오. 별도의 명령이 될 것입니다.

### 섹션 요구 사항

-   **필수 섹션**: 모든 기능에 대해 완료해야 합니다.
-   **선택 섹션**: 기능과 관련이 있는 경우에만 포함합니다.
-   섹션이 적용되지 않는 경우 "해당 없음"으로 남겨두지 말고 완전히 제거합니다.

### AI 생성용

사용자 프롬프트에서 이 사양을 만들 때:

1.  **정보에 입각한 추측하기**: 컨텍스트, 업계 표준 및 일반적인 패턴을 사용하여 격차를 메웁니다.
2.  **가정 문서화**: 가정 섹션에 합리적인 기본값을 기록합니다.
3.  **명확화 제한**: 최대 3개의 [명확화 필요] 마커 - 다음과 같은 중요한 결정에만 사용합니다.
    *   기능 범위나 사용자 경험에 큰 영향을 미칩니다.
    *   서로 다른 의미를 가진 여러 합리적인 해석이 존재합니다.
    *   합리적인 기본값이 없습니다.
4.  **명확화 우선순위 지정**: 범위 > 보안/개인 정보 > 사용자 경험 > 기술적 세부 정보
5.  **테스터처럼 생각하기**: 모든 모호한 요구 사항은 "테스트 가능하고 명확한" 체크리스트 항목에 실패해야 합니다.
6.  **명확화가 필요한 일반적인 영역**(합리적인 기본값이 없는 경우에만):
    *   기능 범위 및 경계(특정 사용 사례 포함/제외)
    - 사용자 유형 및 권한(여러 충돌하는 해석이 가능한 경우)
    - 보안/규정 준수 요구 사항(법적/재정적으로 중요한 경우)

**합리적인 기본값의 예**(이러한 항목에 대해 묻지 마십시오):

- 데이터 보존: 해당 도메인에 대한 업계 표준 관행
- 성능 목표: 지정되지 않은 경우 표준 웹/모바일 앱 기대치
- 오류 처리: 적절한 대체 기능이 있는 사용자 친화적인 메시지
- 인증 방법: 웹 앱의 경우 표준 세션 기반 또는 OAuth2
- 통합 패턴: 지정되지 않은 경우 RESTful API

### 성공 기준 지침

성공 기준은 다음과 같아야 합니다.

1.  **측정 가능**: 특정 메트릭(시간, 백분율, 개수, 비율) 포함
2.  **기술에 구애받지 않음**: 프레임워크, 언어, 데이터베이스 또는 도구에 대한 언급 없음
3.  **사용자 중심**: 시스템 내부가 아닌 사용자/비즈니스 관점에서 결과 설명
4.  **검증 가능**: 구현 세부 정보를 알지 못해도 테스트/검증 가능

**좋은 예**:

- "사용자는 3분 이내에 결제를 완료할 수 있습니다"
- "시스템은 10,000명의 동시 사용자를 지원합니다"
- "검색의 95%가 1초 이내에 결과를 반환합니다"
- "작업 완료율이 40% 향상됩니다"

**나쁜 예**(구현 중심):

- "API 응답 시간은 200ms 미만입니다"(너무 기술적이며 "사용자가 즉시 결과를 볼 수 있습니다" 사용)
- "데이터베이스는 1000 TPS를 처리할 수 있습니다"(구현 세부 정보, 사용자 중심 메트릭 사용)
- "React 구성 요소가 효율적으로 렌더링됩니다"(프레임워크별)
- "Redis 캐시 적중률 80% 이상"(기술별)
