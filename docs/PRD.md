# cork — 컨테이너 횡단 터미널 Todo CLI PRD

**Date**: 2026-07-19 | **Author**: beomeodev | **Status**: Draft

## Executive Summary

여러 격리 환경(컨테이너, VM, 워크트리)을 오가며 일하는 개발자와 그 환경들 안에서
동작하는 AI 에이전트를 위해, 공유 파일 하나를 단일 진실로 삼는 초경량 터미널 todo
CLI `cork`를 만든다(Go 단일 정적 바이너리 — 어느 환경이든 추가 설치 없이 실행). 어느 환경에서 등록하든 모든 환경에서 같은 목록이 보이고, 각
항목이 "어느 환경에서 할 일인지"를 명시하므로, 환경별로 반복해야 하는 후속 작업의
누락 사고가 사라진다. 오픈소스로 공개한다.

## 1. Problem

- **Who**: 여러 프로젝트를 각각 독립 컨테이너(또는 VM, 워크트리)에서 운영하는
  개발자, 그리고 각 환경에서 작업하며 할 일을 남겨야 하는 AI 코딩 에이전트.
- **What**: 환경 간에 공유되는 할 일 목록이 없다. 한 환경에서 발생한 후속 작업
  ("이 컨테이너에서 `git pull` + `make up` 할 것")을 다른 환경이나 나중의 자신에게
  전달할 표준 통로가 없어, 사람의 기억에 의존한다.
- **Why painful**: 후속 작업 누락은 조용히 곪는다. 워크플로우 배포 후 개별
  컨테이너의 갱신 절차를 잊으면 구버전 도구로 계속 작업하게 되고, 문제를 인지하는
  시점은 이미 잘못된 결과가 나온 뒤다.
- **Evidence**: 본 저장소 운영 실측 — SPECTER 워크플로우를 8~9개 프로젝트에
  방송(`/ms.sync`)한 뒤 컨테이너별 후속 절차(`git pull`, `make up`, 바이너리
  재설치)를 누락해 구버전으로 작업한 사고가 반복 발생(2026-07 `np` 구버전 바이너리
  사고는 동일 원인으로 2회 재발). 기존 대안(Discord 봇, GUI 오케스트레이터)은
  터미널 중심 워크플로우와 결이 다르고, Taskwarrior류는 환경마다 바이너리 설치가
  필요해 격리 환경 다수 운영과 맞지 않았다.

## 2. Personas

- **Primary — 멀티 환경 개발자**: 여러 컨테이너 프로젝트를 병행 운영하는 개인
  개발자. 어느 터미널에서든 명령 하나로 전체 할 일을 확인하고 조작하려 한다.
- **Secondary — AI 코딩 에이전트**: 각 환경 안에서 동작하며, 사람 개입 없이 할 일을
  등록·조회한다(예: 배포 도구가 방송 후 대상 환경마다 후속 작업을 등록). 기계가
  다루기 쉬운 저장 형식과 비대화식 CLI 호출이 필요하다.
- **Tertiary — 오픈소스 사용자**: 공유 파일시스템(호스트 마운트, NFS 등)을 가진
  임의의 멀티 환경 구성에서 같은 문제를 겪는 외부 사용자.

## 3. Strategic Context

- 개인 워크플로우 도구를 범용화해 오픈소스로 공개하는 첫 번째 독립 패키지.
- 멀티 에이전트·멀티 환경 운영이 보편화되면서 "환경 간 얇은 조율 통로"의 수요가
  커지는 시점이다. 무거운 오케스트레이션 플랫폼과 달리 파일 하나 + CLI 하나로
  끝나는 최소 해법으로 차별화한다.

## 4. Solution

핵심 설계 원칙: **단일 공유 파일이 단일 진실**, **런타임 의존성 제로(Go 단일
정적 바이너리)**, **특정 워크플로우(SPECTER 포함)에 비의존**.

주요 기능:

1. **저장소**: 설정 가능한 데이터 디렉터리(환경변수 우선, 합리적 기본값) 아래
   JSONL 한 파일. 모든 환경이 같은 경로를 마운트하면 그 즉시 공유 목록이 된다.
2. **워크스페이스 컬럼**: 모든 항목은 대상 워크스페이스(컨테이너/프로젝트
   식별자)를 가진다. 목록은 워크스페이스별 그룹·필터가 기본이다. 사용자 노출
   용어는 `workspace`로 확정(2026-07-19).
3. **워크스페이스 레지스트리**: `cork`가 자체 관리하는 워크스페이스 목록(추가/삭제/조회). 외부
   워크플로우의 레지스트리를 읽지 않는다.
4. **팬아웃 등록**: 한 번의 명령으로 등록된 모든 워크스페이스에 같은 내용의 항목을
   개별 생성(워크스페이스만 다름). 각 사본의 완료는 서로 독립이다.
5. **기본 조작**: add / list / done / undo / edit / rm. 기본 목록은 미완료만
   표시하고, 완료분은 별도 뷰로 조회한다.
6. **우선순위 핀**: 단일 플래그로 항목을 목록 상단에 고정(등급제 없음).
7. **검색·필터**: 텍스트 검색과 워크스페이스 필터.
8. **출처 기록**: 등록 주체(사람/에이전트 식별자)와 등록 시각을 자동 기록.
9. **인터랙티브 TUI**: 인자 없이 실행하면 커서 이동(↑↓), Space 완료 토글,
   `e` 편집, `d` 삭제(확인 후), Tab 활성/완료 전환, `/` 검색, 워크스페이스 필터, `q`
   종료를 지원하는 전체 화면 목록. 마우스 클릭 토글은 지원 터미널에 한해
   best-effort로 제공한다.
10. **동시 쓰기 보호**: 파일 잠금으로 여러 환경의 동시 저장 시 유실을 방지한다.
    동시 수정의 사용자 가시 규칙은 **항목 단위 최종 저장 승리(last-write-wins)** 로
    정의한다 — 서로 다른 항목에 대한 동시 변경은 모두 보존되고, 같은 항목에 대한
    동시 변경은 나중에 저장된 쪽이 남는다.
11. **워크스페이스 자동 감지**: `-w`(워크스페이스 지정) 생략 시 환경변수 → git 저장소명 →
    현재 디렉터리명 순으로 감지한다.
12. **완료분 자동 보관**: 완료 후 일정 기간이 지난 항목은 보관 파일로 이동해
    본 파일 비대화를 막는다. 보관분도 플래그로 조회 가능하다.
    🔶 Assumption: 기간 기본값 30일(런타임 조정 가능 설정).
13. **배포 형태**: Go 단일 정적 바이너리. GitHub Release 바이너리, `go install`,
    Homebrew 설치를 지원하고, 공유 마운트에 바이너리 하나를 놓고 전 환경에서
    실행하는 배치도 지원한다(환경별 설치 불필요 — 단, 동일 CPU 아키텍처 전제,
    Risks 참조).
14. **고유 ID**: 모든 항목은 목록 표시 순번과 무관한 고유 ID를 가지며, 조작
    (done/undo/edit/rm)은 고유 ID 기준이다(입력 편의를 위한 단축 표기 허용).
    환경마다 목록 표시 순서가 달라도 같은 ID는 항상 같은 항목을 가리킨다.
15. **파일 포맷 버전**: 데이터 파일에 포맷 버전을 기록한다. 서로 다른 버전의
    `cork`가 같은 파일을 만질 수 있으므로(설치본과 공유 마운트 바이너리 혼용),
    구버전은 자신이 모르는 필드를 삭제하지 않고 보존한다.
16. **마운트 실패 시 시끄러운 실패**: 데이터 디렉터리를 찾을 수 없거나 접근할 수
    없으면 명시적 오류로 종료한다. 절대로 그 자리에 새 빈 데이터 파일을 조용히
    만들지 않는다(빈 목록을 정상 상태로 오인시키는 조용한 폴백 금지).
17. **기계 판독 출력**: `--json` 플래그로 조회·조작 결과를 JSON으로 출력한다
    (에이전트 페르소나용 안정 계약; 사람용 출력 문구 변경이 에이전트를 깨지 않게).
18. **텔레메트리 없음**: 네트워크 통신을 일절 하지 않는다. 모든 데이터는 사용자가
    지정한 로컬/마운트 경로에만 존재한다.
19. **지원 플랫폼**: 1차 지원은 Linux/macOS. Windows는 Go 크로스 컴파일로 기술적
    제약이 없으므로 TUI 포함 2차 범위에서 검증 후 지원한다(1차에서는 검증 대상만
    제외, 빌드는 제공).

SPECTER 연동(방송 후 자동 등록, 세션 시작 훅 알림)은 `cork` 본체가 아니라 SPECTER
저장소 쪽의 얇은 어댑터로 구현한다 — 8. Out of Scope 참조.

## 5. Success Metrics

| Metric | Current | Target |
|--------|---------|--------|
| 방송 후 환경별 후속 작업 누락 사고(운영 실측) | 반복 발생 | 0 |
| 할 일 등록~타 환경 조회까지 필요한 수동 절차 | 기억/메모에 의존 | 명령 1회 |
| **Guardrail: 동시 쓰기 시 데이터 유실** | — | 0건 (잠금 계약 위반 즉시 결함) |
| **Guardrail: 사용자 측 런타임 요구사항** | — | 0 유지 (단일 정적 바이너리 외에 아무것도 요구하지 않음) |

측정 방법: 첫 지표는 도입 후 4주간, 방송(`/ms.sync`) 이벤트마다 대상 환경별 후속
작업의 완료 여부를 `cork`의 완료 기록으로 집계한다(팬아웃 항목이 곧 측정 단위).
누락 사고 = 다음 방송 시점까지 미완료로 남은 후속 작업 항목.

## 6. User Stories & Acceptance Criteria

1. As a 멀티 환경 개발자, I want 어느 환경에서든 같은 목록을 보고 조작하기를,
   so that 환경 이동이 할 일 파악을 방해하지 않는다.
   - When 서로 다른 환경에서 같은 데이터 디렉터리를 바라보고 `cork` 목록을 실행하면,
     the CLI shall 동일한 항목 집합을 표시한다.
   - When 어느 환경에서든 고유 ID로 항목을 조작하면, the CLI shall 목록 표시
     순서와 무관하게 해당 ID의 항목에만 조작을 적용한다.
   - When 데이터 디렉터리를 찾을 수 없거나 접근할 수 없으면, the CLI shall
     명시적 오류로 종료하고 새 데이터 파일을 생성하지 않는다.
2. As a 개발자, I want 워크스페이스 생략 등록을, so that 현재 프로젝트의 할 일을 빠르게
   남길 수 있다.
   - When 워크스페이스 지정 없이 항목을 추가하면, the CLI shall 환경변수 → git 저장소명
     → 디렉터리명 순으로 감지한 워크스페이스를 부여한다.
3. As a 개발자, I want 팬아웃 등록을, so that 모든 환경에서 반복해야 하는 작업을
   한 번에 배포할 수 있다.
   - When 팬아웃 옵션으로 항목을 추가하면, the CLI shall 레지스트리의 워크스페이스
     수만큼 동일 내용·상이 워크스페이스의 독립 항목을 생성한다.
   - When 팬아웃으로 생성된 한 사본을 완료 처리하면, the CLI shall 다른 워크스페이스의
     사본 상태를 변경하지 않는다.
4. As a 개발자, I want 완료 토글과 완료 목록 조회를, so that 실수 복구와 이력
   확인이 가능하다.
   - When 완료된 항목에 완료 해제를 실행하면, the CLI shall 해당 항목을 미완료
     상태로 되돌린다.
   - When 완료 뷰를 요청하면, the CLI shall 완료 항목(보관분 포함 여부는 플래그)을
     표시한다.
5. As a 개발자, I want 편집·삭제·핀·검색을, so that 목록을 실제 상황에 맞게
   유지할 수 있다.
   - When 항목을 핀 지정하면, the CLI shall 해당 항목을 소속 워크스페이스 그룹 상단에
     고정 표시한다.
   - When 항목을 편집하면, the CLI shall 내용만 갱신하고 고유 ID·출처·시각 기록은
     유지한다.
   - When 항목을 삭제하면, the CLI shall 해당 항목을 목록과 완료 뷰 모두에서
     제거한다.
   - When 텍스트 검색을 실행하면, the CLI shall 검색어를 포함한 항목만 표시한다.
   - When 워크스페이스 필터를 지정하면, the CLI shall 해당 워크스페이스의 항목만 표시한다.
6. As an AI 에이전트, I want 비대화식 등록과 출처 자동 기록을, so that 사람이
   "이건 누가 넣었나"를 추적할 수 있다.
   - When 항목이 추가되면, the CLI shall 등록 주체 식별자와 등록 시각을 항목에
     기록한다.
   - When `--json` 플래그로 조회하거나 조작하면, the CLI shall 기계 판독 가능한
     JSON을 출력한다.
7. As a 개발자, I want 인터랙티브 TUI를, so that 항목 번호를 타이핑하지 않고
   목록을 정리할 수 있다.
   - When TUI에서 항목 위에서 Space를 누르면, the TUI shall 해당 항목의 완료
     상태를 토글하고 화면을 갱신한다.
   - When TUI에서 삭제를 요청하면, the TUI shall 확인 단계를 거친 뒤에만 항목을
     제거한다.
8. As a 개발자, I want 동시 쓰기 안전을, so that 여러 환경·에이전트가 같은 파일을
   써도 데이터가 유실되지 않는다.
   - When 복수 프로세스가 서로 다른 항목을 동시에 변경하면, the storage layer
     shall 잠금으로 직렬화하여 양쪽 변경을 모두 보존한다.
   - When 복수 프로세스가 같은 항목을 동시에 변경하면, the storage layer shall
     나중에 저장된 변경을 최종 상태로 남긴다(항목 단위 last-write-wins).
   - When 신버전 `cork`가 기록한 파일을 구버전 `cork`가 수정하면, the storage layer
     shall 자신이 모르는 필드를 삭제하지 않고 보존한다.
9. As a 개발자, I want 워크스페이스 레지스트리 관리를, so that 팬아웃 대상을 내가
   직접 통제할 수 있다.
   - When 워크스페이스를 등록하면, the CLI shall 이후의 팬아웃 등록 대상에 해당 워크스페이스를
     포함한다.
   - When 워크스페이스를 레지스트리에서 제거하면, the CLI shall 이후 팬아웃에서 제외하되
     그 워크스페이스의 기존 항목은 삭제하지 않는다.

## 8. Out of Scope & Dependencies

제외(각각 대체 목적지 명시):

- **마감일·알림·반복 일정** → 그 수준의 일정 관리가 필요해지면 Taskwarrior 등
  기성 도구로 이관(데이터가 단순 JSONL이라 이사 비용 낮음).
- **하위 작업·의존 관계** → 프로젝트 내부 작업 구조는 각 프로젝트의 워크플로우
  (SPECTER의 Feature Map 등)가 담당.
- **담당자 배정·사용자 권한** → 단일 신뢰 도메인의 공유 파일이라는 설계 철학과
  상충. 팀 규모 요구는 별도 서버형 도구의 영역.
- **동기화 서버·클라우드 백엔드** → 전송 계층은 공유 파일시스템 그 자체. 마운트를
  공유할 수 없는 구성은 지원 대상이 아니다.
- **SPECTER 연동(방송 후 자동 등록, 세션 시작 훅 알림)** → SPECTER 저장소 측
  후속 작업으로 분리. `cork` 본체에는 SPECTER 의존 코드를 두지 않는다.

의존/전제:

- 모든 대상 환경이 같은 데이터 디렉터리를 마운트할 수 있어야 한다.
- 파일 잠금이 유효한 파일시스템이어야 한다(아래 Risks 참조).

## Risks

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|-----------|
| 네트워크 파일시스템(NFS 등)에서 파일 잠금 신뢰성 저하 | 중 | 데이터 유실 | 잠금 + 원자적 임시파일 교체(atomic rename) 병행; 지원 파일시스템 범위를 문서에 명시 |
| 공유 마운트 단일 바이너리 배치가 CPU 아키텍처를 탐(호스트/컨테이너 아키텍처 혼재 시 한 바이너리로 커버 불가) | 중 | 배치 실패 | 아키텍처별 바이너리 동시 제공; 필요 시 아키텍처를 감지해 실행하는 얇은 셸 래퍼 제공 |
| 단일 파일 성장에 따른 성능 저하 | 저 | 체감 지연 | 완료분 자동 보관(기능 12)으로 본 파일 크기 상한 유지 |
